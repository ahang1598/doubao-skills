# 输出与复核

本文件只管“怎么交付、交付前检查什么”。非默认格式路由见 `output-format-routing.md`。

## 默认输出

默认创建飞书文档。只要用户没有明确表达预期格式，就必须创建飞书文档；不要用聊天正文、Markdown、表格、Word 或 PPT 代替默认交付。只有用户明确要求 Markdown、表格、PPT、Word/docx 或不要创建文档时，才按用户指定格式输出；其中 Word/docx 必须通过 `lark-doc` Skill 的 `office-word` 子模块生成真正 `.docx`。

若用户明确只要精简回答、概括、小结、300 字以内或一段话，不进入完整文献检索交付。仅做 1-2 个高价值批次的轻量检索，以普通聊天直接回答，并保留关键来源链接；不创建飞书文档、PPT、Word 或 CSV。用户只是明确要求 Markdown、表格或不要创建文档，但仍要求完整主题检索、综述或调研时，仍按完整流程创建 CSV，并以用户指定格式交付。

聊天窗口只用于短状态和最终入口，不承载完整报告、文献台账、证据表草稿、检索流水、大量文献列举、候选文献清单或完整参考文献展开。这些内容必须放入飞书文档/目标产物正文、附录或本地 CSV。

默认飞书文档的创建顺序：

1. 确认用户没有明确要求非飞书格式，将 `target_format` 视为 `feishu_doc`。
2. 加载 `assets/feishu-doc-template.xml`，把结论、检索范围与精简检索方法、证据图组、证据表和参考文献写入飞书 XML；写正文前先从 CSV 建立 `evidence_id -> anchor_text -> url` 引用地图。不要另写“简洁 XML”绕过模板中的图表章节。
3. 运行 `scripts/sanitize-feishu-report.js` 或等价清理，剥离任何 HTML 上标标签和旧模板遗留的可见“要求/写法/链接要求”说明；再运行 `scripts/validate-report-whiteboards.js` 或等价检查，确保 XML 不含 HTML 上标标签，并且包含论文关系/核心逻辑梳理 Mermaid 白板图、关键研究里程碑时间线图、正文引用来源就地可点击、文末参考文献为 `标题｜来源｜类型｜年份｜链接` 五列表格且链接列可点击、无内部检索工具名、无可见模板指令。只要校验仍提示存在上标标签、模板指令、缺少主图、白板数量不足、正文来源链接风险、参考文献表格列不合格或内部工具名，就必须回到 XML 草稿修复，不能调用 `lark-doc` Skill。
4. 使用已有 `lark-doc` Skill 创建或更新飞书文档，取得文档链接、文档标识或可打开入口。
5. 取得文档 ID 后，交付 CSV 附件：先进入 CSV 所在目录，再用相对路径把同一个 CSV 作为文件附件插入该飞书文档末尾。附件类型必须设为 file；命令形态使用 `--type file`，参数/JSON 形态必须写成 `"type": "file"`。`--file` 必须是相对路径，不要把绝对路径传给插入命令。
6. CSV 附件插入成功后，在文档末尾追加一句普通正文：`完整文献检索记录见表格。`
7. CSV 附件只作为文档末尾文件块交付；不要把 CSV 全量内容、预览表或交付检查表展开进飞书正文。
8. 只有完成第 4-7 步后，默认任务才能进入最终回复。若第 5 步失败，不能伪装成已插入附件；最终回复可用一句话说明附件插入失败，但不要列出本地 CSV、文献台账或台账目录。

如果第 4 步失败或当前环境没有 `lark-doc`，最终回复必须写 `飞书文档创建失败`，说明原因，并给出可重试飞书 XML 草稿文件路径或完整 XML 草稿入口。不能只交付 CSV、PMID 列表或聊天正文。

## CSV 附件交付

默认飞书文档完成后，原始文献台账 CSV 只作为飞书文档末尾文件附件交付；附件用于下载完整循证检索记录，不用于替代正文参考文献表和正文来源链接。最终回复不再单独列出本地 CSV、文献台账或台账目录。

操作顺序：

1. 确认本地 CSV 已通过校验，且记录 `csv_directory_absolute_path`、`csv_file_name`、`feishu_doc_id`。
2. 进入 CSV 所在目录。
3. 仍在 CSV 所在目录，把 CSV 插入飞书文档作为文件附件；附件类型必须设为 file，参数/JSON 形态必须写成 `"type": "file"`。
4. 附件插入成功后，用 `lark-doc` 的追加/插入能力在文档末尾追加普通段落：`完整文献检索记录见表格。`

命令模式：

```bash
cd "{{csv_directory_absolute_path}}"
lark-cli docs +media-insert --doc {{feishu_doc_id}} --file "./{{csv_file_name}}" --type file
```

注意：`--file` 必须使用 `./{{csv_file_name}}` 这种相对路径；不要用 `/absolute/path/to/file.csv`。`--type file` 不能省略；若调用方使用结构化参数而不是命令行，必须传入 `"type": "file"`，不要传 `image` 或其他媒体类型。不要在最终文档正文或最终回复里展示上述命令、插入过程或内部工具名。附件插入成功后，文档末尾必须出现固定句子：`完整文献检索记录见表格。` 附件插入成功时，最终回复可以写：`完整的循证检索记录见文档末尾附录。`

默认飞书文档结构：

1. 结论
2. 检索问题与范围（含精简检索方法与关键词）
3. 证据脉络与研究关系：论文关系梳理、关键里程碑
4. 指南与专家共识
5. 系统综述与关键研究
6. 最新进展、热点、争议与空白
7. 学习、课题或论文写作建议
8. 完整参考文献
9. 免责声明

为避免长文档一次性写入超时，默认飞书文档按 3 个 section 规划和写入：

| section | 包含章节 | 写入方式 |
|---|---|---|
| `section-1` | 标题、1 结论、2 检索问题与范围、3 证据脉络与研究关系 | 先用 `lark-doc` 创建文档骨架和首段内容 |
| `section-2` | 4 指南与专家共识、5 系统综述与关键研究 | 用 `lark-doc` 的追加或插入能力接在 `section-1` 后 |
| `section-3` | 6 最新进展/热点/争议/空白、7 学习/课题/论文写作建议、8 完整参考文献、9 免责声明 | 用 `lark-doc` 的追加或插入能力最后写入 |

执行时先在本地生成完整 XML 草稿并运行校验；校验通过后再把同一份内容拆成 3 个 XML section 依次写入。不要为了分段写入跳过完整校验，也不要把 3 段拆成 3 个独立文档。若追加或插入失败，保存对应 section XML 文件，修复后从失败 section 重试。

可按任务增删章节，但必须保留：结论、检索问题/范围/精简检索方法、证据脉络、系统综述/关键研究逐篇解读、参考文献。CSV 台账不展开为正文内容；默认飞书文档创建后作为文件附件插入文档末尾，最终回复不单独给出文献台账 CSV 或台账目录链接。

## 强制样式

默认飞书文档必须图文并茂，但不牺牲准确性。完整报告至少包含：

- 结论章节：先直接回答用户问题，callout 标题用“总结”，并优先用无序列表呈现 `检索结论`、`实践启示`、`研究局限和空白`；可用加粗、浅黄/浅绿 callout、表格单元底色或语义色突出重点，不强制表格。
- 检索问题表：主题、PICO/PCC/论文论点、纳入排除范围，并用 3-5 行精简展示检索来源、核心关键词、检索批次/效率和证据覆盖局限。
- 精简固定可视化组：默认插入 `论文关系/核心逻辑梳理 Mermaid 白板图 + 关键研究里程碑时间线图`。关系图展示指南、综述、关键研究、最新进展、安全/监管证据如何支持、限制或补充结论，可用多样十六进制语义色区分节点类型；时间线图展示领域演进和关键研究贡献，优先采用“横向年份/阶段轴 + 时间节点下方纵向研究卡片列”的标准时间轴样式。
- 指南与专家共识、系统综述、关键研究和最新进展章节优先用“每篇/每份文献一个小节”的段落解读，说明主要结论、适用边界、局限和引用用途；不默认压成大表。
- 学习、课题或论文写作建议按用户用途自由组织，优先用无序列表分点呈现学习路径、课题机会、论文引用用途和引用风险；可用加粗、浅黄提示或语义色突出重点，不要默认约束为表格。
- 任务相关补充内容优先用正文段落或短列表；争议、研究空白、引用逻辑、学习路径或指南差异确实需要横向比较时，才补充标准表格。只有用户明确要求，或确实能稳定映射为飞书标准图型时，才额外生成图。
- 检索方法不再单独成章；精简并入“检索问题与范围”，只保留外部来源、核心关键词、检索批次/效率、纳入排除和证据覆盖局限。不要铺开完整检索流水或大量关键词组合。

语义色建议：

| 用途 | 颜色 |
|---|---|
| 核心结论/证据方向一致 | `light-green` |
| 检索范围/背景说明 | `light-blue` |
| 证据有限/待核验/需谨慎 | `light-yellow` |
| 证据冲突/争议 | `light-orange` |
| 安全警示/不支持 | `light-red` |

不要生成伪医学影像、伪数据可视化或难以在飞书稳定显示的复杂 diagram。热点主题表若基于本次检索归纳，必须说明不等同于完整文献计量分析。

## 飞书写法

优先使用：`<title>`、`<h1>`、`<h2>`、`<callout>`、`<table>`、`<ul>`、`<ol>`、`<a href="https://...">来源标题</a>`、`<whiteboard type="mermaid">`。

禁止在飞书文档 XML 中使用 HTML 上标标签。不要用上标标签做引用编号、脚注或证据编号；上标包裹的引用编号必须改成普通文本 `[12]`、`[G1]`、可点击文献名或文末参考文献表。

正文所有被引用的文献名本身必须带超链接；首次出现时标注来源/期刊、年份和级别。推荐写法：`[G1] <a href="{{url}}">{{文献名}}</a>（{{来源/期刊}}，{{年份}}，{{来源级别}}）`。参考文献表不能替代正文链接，不能只写 `[G1]`、`[12]`、PMID、DOI 或纯文本题名。文献题名清单、候选文献列表和编号参考文献列表也必须逐条链接题名；`1. 文献题名。期刊，年份。` 这种无链接列表视为交付失败。

正文不能出现无链接来源小标题。以下写法不合格：

`《某某疾病诊疗指南（2025）》`

`DAPA-CKD 研究显示...`

正确写法是把来源名本身做成链接：

`[G1] <a href="{{url}}">某某疾病诊疗指南（2025）</a>（{{来源/期刊}}，2025，A 级指南）`

`<a href="{{url}}">DAPA-CKD</a>（NEJM，2020，A 级来源/高质量 RCT）显示...`

表格建议：

- 检索问题表：主题、框架、范围、结局、时间、语言、纳入/排除。
- 指南对比表：仅在需要横向比较多份指南差异时使用，包含指南、机构/年份、推荐、强度/证据等级、适用范围、更新点；单纯“指南与专家共识”章节默认用逐条解读。
- 关键研究解读：每篇文献一个小节，题名或研究简称可点击，并解释主要结论、局限和引用用途。
- 热点表：主题簇、代表文献、近年变化、证据强度、研究空白。
- 论文引用对照表：仅在用户明确需要论文写作引用安排时补充；学习、课题或论文写作建议章节默认不用表格。

Mermaid 白板必须遵守 `synthesis-and-visualization.md` 中的安全规则。

## 参考文献

文末参考文献分组：

1. 核心证据：已读全文、指南正文或官方页面，支撑关键结论。
2. 辅助参考：仅摘要、背景资料、教材/实践材料或研究线索。
3. 排除/待核验摘要：只在需要透明展示筛选时列代表性记录。

默认飞书文档的完整参考文献必须用表格承载，不用纯编号列表承载。除非用户明确要求特定参考文献格式，文末参考文献表固定为五列：`标题｜来源｜类型｜年份｜链接`。

- `标题`：完整题名、指南名、试验名、教材/监管来源名；不要只写 evidence_id。
- `来源`：期刊、机构、数据库、监管机构或教材来源。
- `类型`：指南、共识、系统综述、Meta、RCT、队列、诊断研究、真实世界研究、安全监管、教材/实践资料、摘要/辅助参考等。
- `年份`：发表、发布或更新年份；不清楚写 `未返回`。
- `链接`：必须是可点击锚文本，如 `<a href="{{url}}">原文/摘要</a>` 或 `[原文/摘要]({{url}})`；不要写裸 URL、DOI、PMID 或“链接未返回”。

无论采用 Vancouver、AMA、APA、GB/T 7714、Markdown、Word、PPT 或飞书格式，文末参考文献表的每一条正式参考文献都必须包含可点击的原文、摘要页、DOI、PubMed、期刊页、指南官网或官方入口链接之一；没有任何可点击入口的条目不得进入正式参考文献，只能保留在 CSV 的待核验/排除记录中。文末有链接不代表正文可以没有链接；正文提到该文献时仍必须就地给出可点击标题或简称。交付前要抽查参考文献章节：每条参考文献行是否有可点击链接；只有编号变蓝、题名和链接列不可点击，不合格。

## 检索问题与范围中的检索披露

检索方法不单独设置一级章节，必须精简并入“检索问题与范围”。至少包含：

- 检索来源/数据库：PubMed/PMC、Cochrane、指南官网、学会官网、监管机构、期刊官网、ClinicalTrials.gov、CNKI/万方/中文期刊官网等实际使用来源；不要展示内部工具名。
- 核心关键词：3-8 个关键中英文词、缩写或 MeSH/Emtree 近似词；不要铺开全部检索式。
- 检索效率/上下文预算：本次用了多少检索批次、Fetch/原文阅读只读取哪些关键部分、为什么没有继续扩大检索。
- 纳入排除与覆盖局限：文献类型、人群/主题、时间范围、语言/地域、未能访问或证据不足的部分。

不要在飞书文档或其他文档类产物中展开 CSV 全量内容、CSV 预览表或 CSV 交付检查表。默认飞书文档应在文末插入 CSV 文件附件；最终回复不单独列出本地 CSV、文献台账或台账目录。

最终回复不得单独提供原始 CSV、本地 CSV 路径、文献台账或台账目录。附件插入成功时，可以在关键说明中写：`完整的循证检索记录见文档末尾附录。`

## 中间消息纪律

允许的中间消息：

- `正在检索和阅读核心文献，完成后会生成专业文档。`
- `核心证据已筛选完，正在整理证据脉络和参考文献。`
- `目标格式创建遇到错误，正在改用可重试模板兜底。`

禁止：

- 不输出完整检索流水、台账、证据表草稿或长篇 todo。
- 不在聊天窗口输出大量文献列举、候选文献清单、筛选记录、证据矩阵或完整参考文献表；这些内容放入文档附录和 CSV。
- 不在创建目标产物前把完整报告先贴到聊天窗口。
- 不把工具返回内容直接整理成用户可见正文。

## 最终回复格式

最终回复保持简短：

1. 核心要点回答：先根据用户提问、调研主题和实际用途，用 2-4 句话或 1-3 条要点直接回答问题；可概括当前证据方向、关键文献类型、主要争议、研究空白或对学习/论文/课题的使用价值。不要用“详见文档”代替回答。
2. 产物入口：`[{{飞书文档/lark-ppt 幻灯片/Word 文件标题}}]({{https_url_or_absolute_local_file_path}})`；如果产物系统只返回文档标识而非 URL，也必须写清可打开入口，不能只写标题。
3. 关键说明：1-3 条，每条一句话；附件插入成功时可写“完整的循证检索记录见文档末尾附录。”
4. 继续调研引导：结合本主题给出 1-3 个相关主题、争议方向或人群/方法学分层，并追问用户是否需要继续调研。
5. 若失败：明确写 `{{产物类型}}创建失败`，说明原因，并提供可重试飞书 XML、PPT XML、供 `lark-doc` Skill 的 `office-word` 子模块重试的 Word 草稿或完整 fallback 内容。

默认模式下，最终回复必须先有核心要点回答，再给飞书文档入口；飞书文档入口只要有 URL 或本地文件路径就必须写成 Markdown 可点击链接。最终回复不得单独列出 CSV、文献台账或台账目录。若只有 CSV、文献列表、PMID 清单、Markdown 草稿、纯标题或“已检索到若干篇”，最终回复无效。

发送最终回复前，按顺序做硬检查：

1. 最终回复开头是否根据用户提问、调研主题和实际用途，直接给出 2-4 句话或 1-3 条核心要点回答。
2. 最终回复是否有目标产物入口；如有 URL 或本地文件路径，是否写成 `[标题](链接或绝对路径)`。
3. 最终回复是否避免单独列出本地 CSV、文献台账或台账目录。
4. 默认飞书文档是否已把 CSV 作为文件附件插入，且插入类型为 file（结构化参数为 `"type": "file"`）；若未完成，最终回复是否明确说明附件插入失败。
5. 目标产物的正文题名清单、证据表、图例和文末参考文献是否每条题名都带链接。
6. 最终回复是否在不喧宾夺主的前提下，给出继续调研相关主题的引导和 1-3 个贴合当前主题的推荐方向。

任一项为否，不发送最终回复，回到修复步骤。

## 最小 QA

交付前检查：

- 已生成目标产物，或已明确失败原因和可重试草稿。
- 最终回复开头已根据用户提问、调研主题和实际用途，用几句话或 1-3 条核心要点直接回答用户问题；没有把文档链接、过程说明或“详见文档”放在回答前面。
- 用户未明确格式时，目标产物是通过 `lark-doc` 创建或更新的飞书文档；最终回复包含飞书文档链接、文档标识或可打开入口。未创建飞书文档且无明确失败兜底时不算完成。
- CSV-only、聊天正文-only、文献清单-only 或 PMID 列表-only 不算默认交付完成。
- 聊天窗口没有铺开大量文献列举、候选清单、筛选记录、证据矩阵或完整参考文献表；这些内容已放入文档/附录/CSV。
- 已创建本地 literature-log CSV，且至少运行 `scripts/validate-literature-log.js` 或等价检查；CSV 至少包含 1 条候选文献/来源记录，只有表头的空 CSV 不能交付。
- CSV 写入遵守标准转义；含逗号、换行、英文双引号、长摘要、作者列表或检索式的字段已用英文双引号包裹，字段内英文双引号已双写；抽查记录列数与表头一致，没有因逗号错列。
- 最终回复不单独列出文献台账 CSV、本地 CSV 路径或台账目录。
- 默认飞书文档已将文献台账 CSV 作为文件附件插入文档末尾；插入必须从 CSV 所在目录执行，`--file` 使用相对路径，插入类型必须为 file（结构化参数为 `"type": "file"`）。附件后已追加固定句子 `完整文献检索记录见表格。` 若附件插入失败，最终回复已简短说明。
- 完整产物在“检索问题与范围”章节中精简写明检索来源/数据库、核心关键词、纳入排除标准、证据覆盖情况、检索批次数、Fetch/原文阅读取材范围和停止扩展理由；不再单独设置检索方法/关键词/证据覆盖类一级章节，该部分不出现内部工具名称或 `xx_search`。
- 检索轮次符合上下文预算：轻量回答一般 1-2 个批次，默认完整报告 2-4 个批次，复杂课题一般不超过 5 个批次；超过时已在“检索问题与范围”中精简说明理由。
- Fetch/原文阅读没有把整页网页、整份 PDF 或低相关内容大量塞入上下文；只读取与核心结论、图表节点或正式参考文献相关的摘要、推荐条文、主要结果、关键表格和局限。

- 正文所有被引用的指南、论文、试验、教材或监管来源标题带超链接；首次出现时标注来源/期刊、年份和级别；只写 evidence_id、编号、PMID、DOI 或纯文本题名不合格。
