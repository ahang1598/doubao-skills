# 章节写作

本参考负责论文各章节的实际写作。豆包环境中，不默认依赖长期存在的本地章节文件夹、计划文件或独立分章流程。章节内容可以先作为对话中的可复制 Markdown 源稿产物交付；但若用户未指定输出介质，最终仍应并入单文件 Markdown 源稿，并继续按主入口要求转为飞书云文档。若当前会话缺少飞书登录、权限或云文档能力，必须明确说明“已完成 Markdown 源稿，待转飞书云文档”，不得把对话中的纯文本章节正文视为最终交付完成。

## 一、写作前门控

开始章节写作前，先确认五件事；信息不足时可输出“待确认/待补”，不要编造。

1. **章节角色**：本章在全文中承担什么功能。
2. **输入材料**：已有题目、大纲、文献、数据、图表、实验日志或用户笔记。
3. **模式要求**：Draft Mode（完整初稿）还是 Final Mode（可提交终稿）。直接生成论文时默认 Draft Mode。
4. **证据要求**：是否涉及文献、实验、图表、事实判断或强结论。
5. **交付范围**：本轮写完整章、写某小节、改写已有文字，还是生成单文件完整初稿。

若涉及文献引用，必须先区分中文与英文来源。中文引用仍需来自 browser-task 页面可见内容、导出引用原文或用户提供的可核验材料；英文引用可来自 `scholar_search` 返回的真实元数据与引用格式，也可来自 DOI/出版社页面、arXiv 页面、browser-task 页面可见内容或用户材料。`scholar_search` 可用于正式英文参考文献、文内引用和元数据层判断，但若正文要写方法细节、实验效果或局限边界，仍需补摘要、DOI/出版社页面、arXiv 页面、browser-task 页面可见内容或用户材料。普通搜索工具、CrossRef、Semantic Scholar、OpenAlex 等后台检索/API 元数据仍不得作为已核验证据。进入正文时，文献性判断必须显式挂文内引用标识，不能只有文末参考文献列表而正文没有引用。未指定体例时，中文默认使用 GB/T 7714 顺序编码制 `[1]`、`[2]`；学校/期刊模板要求著者-出版年制、脚注或其他体例时，正文必须改用对应的作者-年份、脚注序号或模板规定形式。

若用户只要求小段润色，可以简化流程；但只要涉及引用、实验、图表或事实判断，仍要执行对应最小核查。

## 二、章节类型路由

| 章节类型 | 写作前动作 |
|---|---|
| 摘要 | 正文主要内容确定后再写；不新增正文没有出现的结果 |
| 引言 / 相关工作 / 文献综述 | 先按 evidence-driven-writing 规范建立对话 evidence map 与 paragraph blueprint；正文中每个文献性判断或综合段都必须显式挂文内引用标识 |
| 方法 / 模型 / 系统设计 | 按输入到输出的数据流写，不只罗列模块名称 |
| 实验结果 | Draft Mode 可生成 `PLANNING DATA` 表格和结果分析初稿；Final Mode 必须先确认真实数据、指标、表格、图像、日志 |
| 讨论 | 解释结果边界和成立条件，不做宣传式拔高 |
| 结论 | 收束真实贡献、限制和后续工作，不复述摘要 |

### 2.1 整篇论文结构路由

直接生成整篇论文时，不默认使用同一套六章结构。先根据论文类型选择结构：

| 类型 | 推荐结构 |
|---|---|
| 技术类期刊论文 | 题目；摘要；关键词；1 Introduction；2 Related Work；3 Proposed Method；4 Experiments and Results；5 Discussion；6 Conclusion；References。引言通常以连续段落推进，不默认拆成大量小节。 |
| 毕业/学位论文 | 摘要；Abstract；第1章 绪论；第2章 相关理论与研究现状；第3章 方法设计；第4章 实验与结果分析；第5章 讨论；第6章 结论与展望；参考文献；致谢（如需要）。 |
| 综述/课程论文 | 按题目和课程要求压缩结构，保留摘要、引言、主体综述、比较分析、结论和参考文献。 |

当用户说“一个文件就行”时，所有章节、图表计划、参考文献和 `paper-writer-state` 放在同一个对话产物中，不创建或暗示长期存在的分章文件。

## 三、对话章节任务卡

中型章节任务开始前，先在对话中建立轻量任务卡。任务卡不是持久文件要求，它用于锁定边界。

```markdown
### 章节任务卡
- 章节名：
- 模式：Draft Mode / Final Mode
- 本章角色：
- 本轮范围：完整章 / 小节 / 改写 / 蓝图
- 可用材料：
- 必须使用的证据 ID：
- 缺失材料：
- 禁止写入正文：提示词、过程说明、AI 来源说明、待替换说明、无依据强结论
- 交付标准：
```

如果用户已经给出清晰大纲，可直接基于大纲生成任务卡并继续，不要为了形式反复询问。

## 四、写作流程

1. **确认章节目标**：用 2-4 句话复述本章要解决的问题。
2. **建立或引用证据**：文献/实验/图表相关章节必须有 evidence map 或明确待补项。
3. **生成段落蓝图**：只规定段落角色和证据，不规定句式。
4. **写连续正文**：优先自然段落，不把要点直接堆成列表；涉及文献性判断时，在句内或句末挂上 `[1]`、`（作者, 年份）` 或脚注等文内引用标识。
5. **两阶段审查**：先查规格，再查质量。
6. **交付章节产物**：输出章节 Markdown 源稿正文、风险说明和 `paper-writer-state` 更新建议；若用户未指定输出介质，后续仍需并入单文件 Markdown 源稿并继续转飞书云文档。
7. **等待用户确认**：用户确认后再继续下一章；若用户要求加速，可简化确认，但不跳过风险说明。

章节正文不得使用 Markdown 加粗、斜体或 emoji 制造重点；例如 `**预训练与迁移学习**`、`__预训练与迁移学习__`、`_预训练与迁移学习_` 应改为普通小节标题或自然段主题句。

摘要、引言和结论是最容易暴露模板腔的位置。写完整篇或完整章节时，从开始生成正文起就不使用“首先、其次、最后、此外、总之、值得注意的是、需要指出的是、具有重要意义”机械推进；摘要不机械罗列改进模块，引言不把“具有重要意义—近年来显著进展—本文贡献如下”作为固定三段式，结论不重复摘要的效果宣传。若目标模板需要列出贡献，也要让每一项对应具体问题、设计动作、证据位置或适用边界。

## 五、两阶段审查

### 5.1 规格审查

| 检查项 | 说明 |
|---|---|
| 章节功能 | 是否完成本章在全文中的功能 |
| 范围匹配 | 是否只写了本轮要求的章节/小节 |
| 结构完整 | 小节层级和段落顺序是否清晰 |
| 证据使用 | 文献、实验、图表是否按任务卡使用 |
| 文内引用 | 文献性判断是否已显式挂上 `[1]`、作者-年份或脚注等正文引用标识；是否出现“正文无引用、只在文末列参考文献” |
| 公式表格 | 公式是否可渲染，表格是否结构化且未粘连 |
| 正文格式 | 是否残留 `**...**`、`__...__`、`_..._`、emoji、提示词或说明性标记 |
| 缺失标注 | 缺文献、缺图、缺日志、缺数据是否明确标注 |

### 5.2 质量审查

| 检查项 | 说明 |
|---|---|
| 去 AI 味 | 是否存在模板句、高频自指、直译腔、价值宣传化 |
| 论证连续 | 段落之间是否有因果、转折、限定或承接关系 |
| 引用真实 | 引用是否能追溯到 browser-task 页面可见内容、导出引用原文或用户提供的可核验材料；是否有候选文献或搜索/API 元数据冒充正文证据 |
| 引用落点 | 文献综述、相关工作、研究现状、引言中的文献句是否已按目标体例显式挂上文内引用标识；文末参考文献是否与正文引用对应 |
| 实验可信 | 结果强度是否匹配数据、图表、日志和实验条件 |
| 终稿状态 | 是否混入提示词、AI 来源说明、计算纠错或待替换说明 |

审查失败时，先修复；无法修复时，把风险写在交付说明和状态块里，不把问题伪装成终稿。

## 六、章节写作规范

### 6.1 反罗列写作

每章必须形成连续论证，而不是把要点平铺为短段落。正文段落应包含因果、转折、承接或限定关系。

完整初稿生成时，章节开头段不要套用固定背景句。优先从本研究对象的具体约束进入，例如数据集、场景、缺陷类型、评价指标或现有方法的局部不足；只有能说明后续方法选择时，才保留行业意义、技术发展或研究热点类背景。

章节结构也要避免八股节奏：引言不写成“宏大意义—传统方法问题—近年来显著进展—本文贡献如下”；方法不机械拆成“整体结构—模块 A—模块 B—模块 C—总结”；实验不逐表朗读数字；讨论不写成“有效性分析—对比优势—应用价值”三段宣传。若这些结构已经出现，至少重构一个章节功能块为具体约束、失败样例、异常结果或边界讨论。

常见段落角色：

- 背景段：场景约束 → 研究矛盾 → 本章承接。
- 文献段：同类研究共同解决的问题 → 代表性证据 → 尚未覆盖的边界；即使改成综合段，也必须保留 `[1-3]`、`（作者, 年份）` 或脚注等文内引用标识。
- 方法段：输入对象 → 处理过程 → 输出形式 → 设计理由。
- 实验段：评价目标 → 对照关系 → 指标含义 → 可接受结论边界。
- 讨论段：结果含义 → 条件解释 → 局限和后续验证。

### 6.2 方法章节

方法章节采用输入到输出 flow：

1. 输入对象：数据形态、样本、特征、约束。
2. 预处理或表示：清洗、编码、划分、标准化。
3. 核心模型/算法：每个模块说明输入、处理、输出、设计理由。
4. 训练或推理流程：公式、算法、参数更新或决策路径。
5. 输出：预测、解释、告警、指标或下游接口。
6. 与实验对应：关键模块应能映射到消融、对照或局限说明。

禁止只写“由三层组成”“包括若干模块”后缺少数据流、公式位置和可复现步骤。

每个方法模块还必须说明“本文决策”：为什么选择该模块、放在网络或流程的哪个位置、针对本文数据或任务的哪类具体问题、对应哪项实验或图表验证。若只能写通用原理，说明它还只是背景介绍，不是本文方法段。

#### 公式输出规范

- 核心模型公式必须使用 LaTeX 数学格式或标准公式编辑器语法，独立成行并建议编号（如式（1））。
- Markdown/纯文本交付中，行间公式统一写为 `$$...$$`，行内变量写为 `$...$`。
- 飞书云文档交付中，优先使用云文档公式块/公式语法；如果只能先输出文本，必须提供可转换 LaTeX 源并说明应转为公式块。
- 禁止在公式中使用 HTML 标签（如 `<sub>`、`<sup>`）或纯文本下标（如 `NQP_{i,t}`、`β_0`）；上下标必须表达为数学格式。
- 幂指数、上下标、分式必须完整写在同一个公式结构内，不得把指数单独拆成一行。
- 公式后的符号说明必须与公式中的系数符号完全一致；例如公式含 `γ_1`、`γ_2` 时，解释中不得写成重复的“γ和γ”。
- 变量定义段落按“被解释变量 / 解释变量 / 机制变量 / 控制变量”等层级分段，每一类单独成段，不与上一段正文粘连。

### 6.3 实验和讨论章节

实验章节不得保留“实验目的、表位、回填模板、讨论提示、请用户替换”等过程性说明。真实结果用数据支撑；Draft Mode 可使用 mock 或 `PLANNING DATA` 形成完整初稿，但表题、表注或状态块必须保留 `PLANNING DATA - replace before submission` 标记，不能写成已验证结论；Final Mode 不允许保留 mock 数据。

讨论章节不要重复摘要或结果表，而要解释：为什么出现该结果、结果在哪些条件下成立、哪些条件下可能失效、还需要哪些补充实验。

#### 表格输出规范

- 出现“一级指标 / 二级指标 / 衡量指标 / 指标属性”等结构化内容时，必须生成标准表格，不得压缩为连续正文。
- 描述性统计必须使用标准表格，每个变量一行，观测值、均值、标准差、最小值、中位数、最大值等统计量分别成列。
- 相关性分析必须输出为矩阵表，变量名作为行列标题；显著性星号紧跟对应系数，不得因换行被拆散。
- 回归结果必须生成标准三线表或结构化表格；每个模型单独一列，变量、系数、括号内统计量、固定效应、样本量和 R² 分行展示。
- 表格结束后另起段落；表题、表注、下一节标题不得与表格内容粘连。
- Draft Mode 不得只写“表 X 待补”。缺真实数据时，生成带表题、列头、单位、TBD/mock 数值和表注的标准表格，并在表注中写明 `PLANNING DATA - replace before submission`。

#### 图占位输出规范

- 缺真实图片时，Draft Mode 使用正式图占位：`图 X-X 待绘制：图题。内容要求：...`。
- 图占位应说明图中需要展示的对象、关键标注、输入输出关系或坐标轴，而不是只写“此处插图”。
- 技术类期刊论文初稿通常至少保留 3-5 个图占位；毕业/学位论文初稿通常至少保留 6-8 个图占位。
- YOLO/CV/缺陷检测论文常用图包括：技术路线图、整体网络结构图、注意力模块结构图、特征融合结构图、数据集样例图、训练曲线/PR 曲线、检测结果对比图和复杂场景示例图。

### 6.4 结论章节

结论只收束正文已经证明的内容：

- 回应研究目的。
- 概括真实贡献。
- 说明限制。
- 给出与本文问题直接相关的后续工作。

不要在结论中新增文献、实验结果、图表或夸大价值判断。

## 七、输出模板

章节交付状态使用以下四类之一：

| 状态 | 含义 |
|---|---|
| DONE | 本轮范围完成，材料和格式风险已说明且无阻塞问题。 |
| DONE_WITH_CONCERNS | 内容可继续使用，但仍有待核验文献、缺图、缺日志、格式或证据风险。 |
| NEEDS_CONTEXT | 缺少必要材料，继续写会导致编造或空泛。 |
| BLOCKED | 缺少用户授权、知网登录、关键数据或模板，必须暂停。 |

### 7.1 开始写作前

```markdown
我将处理「[章节名]」。

本章角色：[说明它在全文中的作用]
本轮范围：[完整章/小节/改写/蓝图]
可用材料：[文献/数据/图表/用户笔记]
待补材料：[如缺图、缺日志、缺真实实验]
文内引用体例：[默认 `[1]` / 作者-年份 / 脚注 / 模板要求]

若以上理解无误，我将按该边界输出章节正文；缺失材料会在正文外标注，不会补写成事实。
```

### 7.2 章节交付后

```markdown
「[章节名]」本轮产物如下。

状态：DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED

## [章节标题]

[章节正文]

### 本轮审查
- 规格审查：通过 / 有风险（说明）
- 质量审查：通过 / 有风险（说明）
- 文内引用：已按目标体例显式挂出 / 有风险（说明）
- 正文格式：无 `**...**`、`__...__`、`_..._`、emoji / 有风险（说明）
- 待补材料：
- 建议下一步：
```

### 7.3 状态块补充

章节写完后，建议更新主技能的 `paper-writer-state`：

```markdown
当前章节/段落：
已完成内容：
待用户确认：
缺失材料：
高风险问题：
下一步：
```

## 八、错误处理

| 情况 | 处理 |
|---|---|
| 没有大纲 | 先输出章节蓝图或最小大纲，让用户确认后再写正文 |
| 前置章节未完成 | 说明依赖关系；可先写独立小节或待补版，不伪装成终稿 |
| 用户要求跳过确认 | 可加速，但章节完成后仍要给风险说明和下一步 |
| 缺文献 | 不写具体文献结论；先输出检索需求或待核验写法 |
| 文末有参考文献但正文未挂引用 | 视为未完成；补齐正文中的 `[1]`、作者-年份或脚注引用标识后再交付 |
| 缺实验材料 | 不写“全面验证”；写数据需求、占位说明或限定结论 |
| 缺图表 | Draft Mode 生成正式图占位和 `PLANNING DATA` 表格；Final Mode 才正文外列为待补 |

## 九、关键原则

- 一次聚焦一个章节或一个明确小节。
- 每个章节产物都要可复制、可审查、可继续。
- 用户未指定输出介质时，章节正文只是中间 Markdown 源稿产物，最终仍需并入单文件 Markdown 并继续转飞书云文档。
