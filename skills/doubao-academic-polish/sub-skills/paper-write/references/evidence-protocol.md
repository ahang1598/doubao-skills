# Evidence Protocol

在写任何涉及引用、实验或事实判断的正文之前，先用这个协议理清你手上有什么证据。

## Evidence Map 模板

整篇论文任务必须输出这张表给用户。单章节任务在脑子里过一遍即可。

```markdown
| ID | 来源 | 层级 | 能支撑的声明 | 不能支撑的声明 | 计划用在 | 风险 |
|---|---|---|---|---|---|---|
| E1 | 用户：实验数据 Table 2 | L1 | 方法细节、BLEU 数字、消融结果 | — | Results, Discussion | 无 |
| E2 | scholar_search: Author (2023) | L3 | "addressed X using Y" | 方法步骤、具体效果 | Related Work | metadata-only |
| E3 | 用户：摘要片段 | L2 | 研究方向、核心发现方向 | 具体数字 | Intro | 不含数字 |
```

**层级说明**：

| 层级 | 来源 | 能支撑 | 不能支撑 |
|---|---|---|---|
| L0 领域常识 | 从业者无需引用即接受，且不含具体数字/人名/量化比较 | 背景铺垫性陈述 | 含数字/人名/比较的任何声明 |
| L1 全文/数据 | 用户给了全文、数据、图表、实验日志 | 任何声明 | — |
| L2 摘要 | 用户给了摘要或 scholar_search 含摘要 | 研究方向、核心发现概述 | 具体数字、方法步骤、实现细节 |
| L3 元数据 | scholar_search 返回的题名/作者/年份 | "X et al. addressed Y" | 方法细节、效果数字、局限分析、方法论描述 |
| L4 模型记忆 | 你"觉得知道"的 | **什么都不能** | **一切** |

**填写规则**：
- 每个来源一行，标明层级
- **"不能支撑的声明" 列必须填**——这一列的作用是提醒你这个来源的边界
- L3 来源在"风险"列标 `metadata-only`
- L4 来源**不要放进 evidence map**——它不是证据

## Provenance Thinking

写作阶段（S3）的核心纪律：写每段正文之前，先在内部想清楚每个事实性声明的证据来源（用户输入 / scholar_search 结果 / L0 常识）。没有来源的主张不写入正文——在规划阶段就排除。

直接写干净的正文，不在生成的文本流中嵌入任何标签。交付的正文中不包含任何方括号占位标签（`[citation needed]`、`[to be verified]`、`[to be specified by authors]` 等）。

未消解的不确定事实应在交付前通过 scholar_search 验证或删除相关主张；用户未提供的细节直接省略。

## 实验证据门控

涉及实验结果、Discussion 或贡献声明时，逐条检查：

| Gate | 检查 | 不过时怎么办 |
|---|---|---|
| D0 数据边界 | 数据来源、样本、划分是否明确？ | 只写实验计划，不写结果 |
| D1 指标口径 | 指标定义、计算方式是否明确？ | 解释指标用途，不给效果判断 |
| D2 对照 | 基线方法、消融对应是否清楚？ | 贡献降级为设计动机 |
| D3 结果证据 | 结果数字来自用户确认的真实实验？ | Draft Mode 用模糊措辞（"the authors report" / "preliminary results suggest"）替代断言语气；Final Mode 不写效果声明 |
| D4 去污染 | 规划数据和真实数据区分清楚？ | 标注所有规划数据 |
| D5 结论边界 | 结论只覆盖当前数据和实验条件？ | 删除或降级泛化结论 |

## 方法-实验追溯

整篇论文交付前，检查每个贡献声明：

```markdown
| 贡献/主张 | 对应方法段 | 对应实验/图表 | 证据状态 |
|---|---|---|---|
| C1: 提出 X 框架 | §3.1 | Table 2 消融 | L1 已确认 |
| C2: Y 机制设计 | §3.2 | — | 无实验支撑 → 降级 |
```

追溯不到实验的贡献 → Draft Mode 标注 "待验证"；Final Mode 降级或移入 Future Work。
