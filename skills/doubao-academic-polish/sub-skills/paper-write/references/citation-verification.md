# 引用幻觉核查：OrganizeAgent 独立核查协议

这是写作 skill 在引用这一层的幻觉控制。它只管引用真不真，不替代整个 S4，也不接管写作流程。

## 为什么要独立核查（第一性原理）

S4 里其余几项审查是同一个模型在同一段上下文里检查自己刚写的东西。这对引用幻觉几乎无效：模型当初凭记忆写出"Smith et al. (2021) 提出了 X"时，它"觉得"这是真的；紧接着让它自查"这条引用可信吗"，它会再确认一遍同样的记忆——生成和检查是同一个认知动作，共享同一套错误。

引用幻觉（编造不存在的文献、张冠李戴的作者年份出处、错误的引用号、把元数据当全文用）是所有幻觉里**最常见、也最可以被机械证伪**的一类：一条文献要么搜得到要么搜不到，元数据要么对得上要么对不上。这正是可以外包给一道独立工序的部分。

`OrganizeAgent` 的价值就在这里：它把"引用核查"从自检变成独立工序——派一个**全新上下文**的核查 agent，它拿不到你当初为什么这么写的推理，只拿到成稿和参考文献表，唯一任务是用 `scholar_search` 把每条引用逐条查一遍。防线因此从"模型对自己诚实"换成"`scholar_search` 可机械验证"，强一个量级。

## 什么时候触发

按任务粒度分档，不是每段都触发（独立核查会增加一次调度和多次检索的成本）。下面四档互斥且穷尽，没有中间地带：

| 场景 | 是否触发独立核查 |
|---|---|
| 整篇论文 / 多章节初稿，或 Final Mode（投稿前 / 终稿 / 可提交） | **必做**（不论引用多少条） |
| 上一行之外，产出累计引用 ≥ 3 条 | **必做** |
| 上两行之外，单段落 / 短章节且累计引用 ≤ 2 条 | 可省略，用 S4 第三步内联文献检查兜底 |
| 无任何引用 | 不触发（没有可核查对象） |

注：阈值取 ≥ 3 而非更高，是因为 intro、related work 等常见章节的引用数很容易超过门槛；落到内联自检那一侧的产出必须是引用极少的轻量片段，否则等于在常见档位放过引用幻觉。**SKILL.md S4 第二步、verification-checklist.md §2.1 与本表如有出入，以本表为准**——本表是触发条件的唯一权威。

## OrganizeAgent 怎么调度

1. **出稿**：paper-write 完成 S3 写作，产出带 `[N]` 引用的正文 + References 表。
2. **派核查 agent**：`OrganizeAgent` 派一个独立的 citation-verify 子任务，**输入只有两样**——
   - 论文正文（含 `[N]` 标注）
   - References 表（每条的作者 / 标题 / 年份 / 出处）

   **不传**写作时的推理过程、evidence map 的内部推导、CER 骨架。核查 agent 必须从零开始判断，不能被写作时的"自信"传染。
3. **逐条核查**：核查 agent 对每条引用执行下方核查，给出状态。
4. **门禁回调**：只要有任何一条处于**未消解**的 `NOT_FOUND` / `METADATA_MISMATCH` / `OVERCLAIM`，门禁不通过，`OrganizeAgent` **回调 paper-write** 做针对性修正（不是从头重写，只改这几条），改完再次进入核查。**收敛条件是"不存在未消解的问题条目"**，不是字面的"全部 VERIFIED"——一条引用要么 `VERIFIED`，要么已按路由消解：假引用（NOT_FOUND）移出 References 并删/改其支撑句、过度引述（OVERCLAIM）降级为 citation-level 措辞、元数据不符（METADATA_MISMATCH）订正、`INCONCLUSIVE` 已挂起交用户。**消解一律通过改写 / 降级 / 删除 / 挂起完成，绝不靠在正文插入 `[to be verified]` / `[citation needed]` 标签**——那些标签在 S5 会被判为流程失败。OVERCLAIM 一旦降级为 citation-level 就不再判 OVERCLAIM，不会死循环。

## 核查 agent 对每条引用做什么

对 References 表里的每一条 `[N]`：

**第一步：查存在性与元数据。**
- 用 `scholar_search` 搜该文献，**至少换 3 种检索式**：①完整标题；②作者 + 关键词 + 年份；③核心关键词。
- 比对：标题搜得到吗？作者、年份、出处（会议 / 期刊）与搜索结果一致吗？
- **区分"搜不到"与"搜不了"**：只有 `scholar_search` 正常返回、且 ≥ 3 种检索式都零命中，才算"搜不到"（→ `NOT_FOUND`）。如果工具报错、超时、限流、或疑似服务异常的系统性空返回，这是"搜不了"，判 `INCONCLUSIVE`，**绝不当成 `NOT_FOUND`**（见状态表）。

**第二步：查正文引述强度（回到 `[N]` 在正文里出现的地方）。**
- 正文写"Author et al. `[N]` addressed Y"——这是 **citation-level**，元数据对得上即可。
- 正文写"`[N]` achieved 92% on benchmark B"或"their three-stage pipeline does Z"——这是 **full-text-level** 声明，`scholar_search` 只返回元数据（L3），不足以支撑这种具体结果或方法细节的引述，判为 `OVERCLAIM`。

## 状态分级与路由

| 状态 | 判定 | 回调后 paper-write 怎么改 |
|---|---|---|
| **VERIFIED** | 搜到，元数据一致，正文引述是 citation-level | 保留 |
| **METADATA_MISMATCH** | 搜到该文献，但作者 / 年份 / 出处对不上 | 用搜索结果订正 References 条目。**但若单一检索后端的结果与多源 / 常识明显冲突（如把经典论文年份标成未来年份），不盲信后端，标 `[to be verified]` 交用户裁决** |
| **NOT_FOUND** | 工具正常返回、且 ≥ 3 种检索式都零命中 | 删除该引用及其支撑的句子，或用 scholar_search 搜索替代文献。**不得降级为 `[citation needed]`，不得当作既成事实保留** |
| **OVERCLAIM** | 文献真实存在，但正文引述了元数据无法支撑的具体结果 / 方法细节 | 降级为 citation-level 措辞（"X et al. addressed Y"），不标 `[to be verified]` |
| **INCONCLUSIVE** | `scholar_search` 报错 / 超时 / 限流 / 疑似服务异常的系统性空返回 | **既不通过也不删内容**；重试若干次仍失败则把该条挂起、交用户裁决 |

`NOT_FOUND` 是最值钱的拦截——这正是"韩国 4G 密度门槛政策""Thaler v. Vidal 错误案号"那类编造在引用层的形态。而 `INCONCLUSIVE` 与 `NOT_FOUND` 的区分是这道工序不出错的关键：**一次基础设施抖动绝不该删掉本来正确的引用**。把"搜不了"误判成"搜不到"，等于让工具故障去破坏真实内容。

## 诚实边界（这道工序能做什么、不能做什么）

**能抓：**
- 编造的文献（`NOT_FOUND`）——核心价值。
- 张冠李戴的作者 / 年份 / 出处（`METADATA_MISMATCH`）。
- 把元数据当全文用的过度引述（`OVERCLAIM`）。

**抓不住（不要假装能）：**
- **文献真实存在、引述也是 citation-level，但被引作者的主张本身是错的**——核查 agent 没有全文，无法判断。这超出引用核查范围，属于 S4 §2.3 证据匹配审查（需要用户提供全文 / 摘要才能查）。
- **非引用类的、似真却无来源可证伪的事实编造**（如某个不存在的制度细节、某个编造的实验数字）——这不是引用核查能解决的，靠 S1–S3 的证据层级（L0–L4）和 provenance thinking（写前想清来源、无来源不写）在写作阶段兜底。

一句话：这道工序把引用的**存在性**（`NOT_FOUND`）换成 `scholar_search` 可机械验证的硬标准；元数据订正与 `OVERCLAIM` 判定仍含模型判断，且检索后端本身可能出错（见 `METADATA_MISMATCH` 的保护条件），不是纯机械。它只加固引用这一层，是 S4 的一个独立子工序，不替代 S4 的其余各项，也不接管 S0–S3 的写作流程。

**一个承载性前提**：这层的有效性建立在 `OrganizeAgent` 确实派出一个**全新上下文**的独立子 agent、且该子 agent 真能调用 `scholar_search`。如果运行时没有真正隔离上下文，独立核查就退化成同上下文自检——而同上下文自检正是本文件开头论证的、对虚构文献几乎无效的那道工序，这层对引用幻觉也就随之失效。交付到真实产品前应确认运行时满足这个前提，不要把这层当成无条件成立的硬保证。

**当 OrganizeAgent 不可用时的降级规则**：如果运行时没有独立上下文隔离（即 OrganizeAgent 不可用或不支持独立子任务），应用以下更严格的生成约束替代事后核查：(1) 只引用本次会话中通过 scholar_search 实际搜到的文献，不引用任何"记得"的文献；(2) 对每条引用，在写入正文前先用 scholar_search 搜索论文标题确认存在；(3) 引用声明限制在 citation-level（"X et al. addressed Y"），不写具体数字或方法细节；(4) 在交付说明中告知用户：本次未执行独立引用核查。
