# 输出格式路由

本文件只管“用户要哪种交付物时，选择什么模板、怎么创建、怎么校验”。默认飞书文档的章节和 QA 见 `output-and-qa.md`。

## 选择规则

| 用户意图 | 默认产物 | 模板 | 适用场景 |
|---|---|---|---|
| 未指定格式、需要完整报告，或明确提及查指南/文献/完整循证 | 飞书文档 | `assets/feishu-doc-template.xml` | 病例讨论、科室内部共享、核心证据解读和附录较多 |
| 特别简单、直接、风险低的临床知识回答，或用户要求简短回复/简版且问题不涉及具体病例、多方案、安全决策、指南冲突或多篇证据支撑 | 聊天快速循证回答 | `assets/final-response-template.md` 的快速循证回答 | 先快速回答并提供来源链接，再询问是否需要完整报告 |
| 说“PPT、汇报、演示、MDT 展示、幻灯片” | PPT/飞书幻灯片 | `assets/ppt-deck-template.xml` | 需要口头汇报、MDT、教学或管理层快速理解 |
| 明确要求“Word、docx、正式报告、归档、投稿/伦理附件” | 可打开的 Word/docx 报告 | `assets/word-report-template.md` + `references/word-docx-generation.md` | 需要正式书面报告、可打印归档、少用交互块 |
| 明确要求 Markdown、表格或不要创建文档 | 按用户要求 | 仍保留引用链接和结构化证据解读 | 临时沟通、快速预览、工具不可用兜底 |

不确定是否需要完整报告时先按飞书文档执行。只有问题特别简单、直接、风险低，或用户明确要求简短且不涉及具体病例、多方案、安全决策、指南冲突或多篇证据支撑时，才按快速循证回答处理。用户只说“简版”但问题仍涉及病例、方案、安全或证据冲突时，仍按完整循证路径执行，只压缩最终文档篇幅和汇报密度。

## 渐进式披露

先完成共通循证判断：病例结构化或临床问题边界、PICOTS/PECO、检索与原文阅读预算、证据评价、结论和风险。`response_mode=full_report` 时创建原始 CSV 记录；`response_mode=quick_answer` 仅用于特别简单、直接、风险低的问题，不强制创建 CSV。只有在准备交付时才加载对应格式模板：

- 飞书文档：加载 `assets/feishu-doc-template.xml`。
- PPT/幻灯片：加载 `assets/ppt-deck-template.xml`，并在生成前先做 slide plan。
- Word/docx：加载 `references/word-docx-generation.md` 和 `assets/word-report-template.md`，优先生成真正 `.docx`；Markdown 只作为工具不可用时的兜底草稿。

完整报告格式都必须包含：结论先行、病例信息、临床问题分析和解答、证据关系/决策逻辑、从 CSV 核心记录生成的证据解读、风险与下一步、完整参考文献（含链接）。临床问题分析和解答必须位于指南/共识和关键研究之前，先用“问题总结”直接回答医生问题，再按实际问题自由组织分析，说明问题本质、病例信息如何改变判断、证据如何支持或限制回答、哪些风险或缺失信息会改变结论；可适度引用后文证据编号或简称。正文首次提到具体来源必须有链接；文末完整参考文献默认只用 `标题｜来源｜链接` 三列。默认飞书文档创建后，直接把 CSV 作为文件附件插入文档末尾；最终回复不单独列出文献台账、CSV 或台账目录。快速循证回答只用于特别简单、直接、风险低的问题，只需包含核心结论、风险边界、可点击来源链接和完整报告引导。

## 飞书文档

创建飞书文档时，必须使用已有 `lark-doc` Skill，并按 `lark-doc` Skill 的文档创建、更新和校验流程执行。

- 默认使用 XML；用户明确要 Markdown 时才用 Markdown。
- 长报告先创建标题和章节骨架，再按章节插入或更新内容，避免一次性塞入导致顺序反转或结构丢失。
- 使用 `<title>`、`<h1>`、`<h2>`、`<callout>`、`<table>`、`<grid>`、`<whiteboard>`、`<a>`、`<bookmark>` 等结构化块。
- 飞书文档 XML 禁止使用上标标签、脚注式引用或角标引用。
- 医学来源必须在正文中可点击，可用普通链接或允许的 citation 样式组件。
- 结论、风险、证据不足用语义色；不要把整篇文档都做成高亮块。
- 正文首次提到文献名时，文献名本身带超链接，并标注期刊/来源和级别；不要把指南/研究名写成无链接小标题。
- 复杂病例至少有 1 个证据关系图；多证据、多方案或安全复杂时至少有 2 个互补图表。
- 取得文档 ID 后，从 CSV 所在目录用相对路径把同一 CSV 作为文件附件插入文档末尾；不要把 CSV 全量内容、预览表或交付检查表展开到正文。

## PPT/飞书幻灯片

创建 PPT/飞书幻灯片时，必须使用已有 `lark-ppt` Skill。PPT 是汇报材料，不是把报告缩小成多页标题加项目符号。创建前先按 `lark-ppt` Skill 的要求完成 slide plan，每页明确 `layout_type`、`visual_focus`、`text_density`、核心信息和引用来源；不要沿用其他幻灯片 Skill 名称或路径。

默认 10-12 页，优先 11 页左右。除非用户明确要求简版、单页汇报或工具失败兜底，slide plan 不得少于 10 页；若证据复杂，优先拆分关键研究、方案对比、风险监测等页面，而不是压缩成少页长段落。

推荐 10-12 页结构：

1. 封面：病例主题、问题、日期。
2. 结论先行：1 条主结论 + 2-3 条关键依据/风险。
3. 病例与 PICOTS/PECO：多源信息和缺失信息。
4. 临床问题分析和解答：先给问题总结，再按实际问题自由组织病例线索、证据回答、风险边界和需补充信息，可引用后文 [G1]/[R1]。
5. 检索覆盖：检索批次、证据类型、纳入/排除概况；完整的循证检索记录见飞书文档末尾附件。
6. 证据关系图：指南、研究、安全证据如何支持/限制结论。
7. 决策逻辑图：当前病例条件下的分支判断。
8. 指南/共识矩阵：推荐方向、等级、适配性。
9. 关键研究详解：研究设计、结果、期刊/来源级别；证据多时可拆成 2 页。
10. 执行方案与病例化落地：把前面的临床问题分析和解答转译为可讨论的执行路径、方案对比和监测重点。
11. 风险、安全性与下一步：禁忌、监测、药物/特殊人群、需补充信息、MDT/药师/检验影像路径、随访节点。
12. 完整参考文献（含链接）：核心证据、辅助参考、排除/待核验记录；默认使用 `标题｜来源｜链接` 三列。完整的循证检索记录作为附件交付。

设计约束：

- 画布按 960 x 540 规划。
- 每页只讲一个主信息；不要整页长段落。
- 默认生成 10-12 页；少于 10 页必须在最终回复或备注里说明用户要求简版或工具受限。
- 4 页以上的 PPT 至少使用 4 种版式，如封面、结论卡、证据图、决策树、矩阵、风险页。
- 证据页和参考文献页必须显示文献简称、期刊/来源、来源级别和可点击原文/摘要链接；链接优先放正文或核心证据解读中，备注只能作为补充，不能替代正文证据链接。参考文献页默认按 `标题｜来源｜链接` 呈现。
- PPT 中的关键证据关系图、决策逻辑图默认优先使用原生 `shape`、`line`、`table` 或原生 `<chart>`；不要把 Mermaid whiteboard 作为唯一主图。
- 只有在确实需要自动布局、并且能做视觉验收时，才使用 `<whiteboard><mermaid><![CDATA[...]]></mermaid></whiteboard>`。Mermaid whiteboard 必须有同页或相邻页表格/形状兜底，避免白板渲染失败后页面只剩图注。
- 结构化数据图表优先使用原生 `<chart>`；必须同时包含 `chartPlotArea` 和 `chartData`，并检查 `dim1` / `dim2` 数量匹配。
- 生成后必须按 `lark-ppt` Skill 回读/校验：页数正确、无空白页、无明显重叠/溢出、不是清一色标题+项目符号、关键引用链接存在。注意回读 XML 不包含 whiteboard 内部 SVG/Mermaid 内容，不能据此声称白板图已验证。

## Word/docx 报告

Word 适合正式、可打印、可归档的报告。生成 Word/docx 时，先读取 `word-docx-generation.md`，再使用已有 `lark-doc` Skill，并按其顶层路由进入 `office-word` 模块生成或编辑 `.docx`。`assets/word-report-template.md` 只作为内容骨架；用户要求 Word/docx 时，默认交付真正可打开的 `.docx` 文件。

写法要求：

- 使用正式编号标题，少用彩色装饰，保留专业表格。
- 保留正文文献超链接、核心证据解读和完整参考文献表格；正文不得出现无链接的来源小标题。CSV 不作为本地链接写入正文或最终回复。
- 图表可用 Mermaid 图片、表格图或静态流程图，但必须附图例表，保证打印后仍能追溯来源。
- Word 版本不强制使用飞书 callout，但结论、风险、证据不足必须在文字和表格中显著标出。
- 默认 A4、2.5 cm 页边距、宋体小四正文、1.5 倍行距、黑体标题、页眉页脚和稳定表格列宽；不要把 Markdown 改名成 `.docx`。
- 附录可比飞书文档更完整，但不要原样粘贴超宽 CSV；按核心纳入、辅助/仅摘要、排除/待核验分组。
- 生成后检查 `.docx` 可打开、标题层级正常、表格不溢出、图表有编号和图例、参考文献完整且每条都有原文/摘要链接。

## 失败兜底

如果目标格式工具不可用：

- 明确写出失败原因。
- 不要把任务结束为“稍后生成”。
- 提供对应模板的完整草稿：飞书文档 XML、PPT XML 或 Word/docx 可重试 Markdown 草稿。
- 最终回复仍保持简短，说明产物位置或可重试草稿位置。
