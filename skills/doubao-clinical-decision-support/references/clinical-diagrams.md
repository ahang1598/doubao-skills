# 循证图表规范

本文件用于决定“画什么图、图里放什么、如何保证图和证据链一致”。图表方法参考 `standard-feishu-diagram` 的图表选择、语义颜色、节点/连线建模和可编辑飞书白板实践：https://github.com/Hanny-YANG/standard-feishu-diagram

## 触发规则

默认飞书报告至少包含 1 个图表，如果合理，尽量提供 2-4 个互补图表：

- 核心证据包含 3 篇以上指南/共识/论文/安全来源。
- 存在 2 个以上候选诊疗方案、检查路径或用药策略。
- 指南之间不一致，或新研究与旧指南存在张力。
- 涉及特殊人群、合并症、禁忌、相互作用、围手术期、肝肾功能异常等复杂安全判断。
- 医生的问题同时包含“证据是否支持”和“临床上怎么走下一步”。

优先组合：`证据关系 Mermaid 白板图 + 决策逻辑 Mermaid 白板图`。若问题有深度，可继续补充 `思维导图`、`风险矩阵` 或 `泳道图`。默认飞书文档优先使用 Mermaid 白板；PPT/幻灯片仍按 `lark-ppt` 规则优先使用原生形状和连线。

## 图表选择

| 场景 | 首选图 | 用途 | 何时追加第二张 |
|---|---|---|---|
| 多篇指南/论文共同支持一个结论 | 证据关系图 | 展示病例问题、指南、研究、安全证据和结论之间的支持/限制/冲突关系 | 同时有多个方案时追加决策树 |
| 诊疗路径、检查顺序、管理流程 | 流程图 | 展示先后步骤、分支、缺失信息和异常处理 | 涉及多角色执行时追加泳道图 |
| 药物/方案是否适用、禁忌筛查 | 决策树 | 展示条件判断、分支和条件性结论 | 证据很多时追加证据关系图 |
| 多学科、院内流程、患者-医生-药师-检验等分工 | 泳道图 | 展示每一步由谁负责、何时交接 | 存在关键阈值时追加流程图或决策树 |
| 指南冲突、新证据尚未进入指南 | 证据冲突图或矩阵 | 展示不同来源的推荐方向、适用人群和不一致点 | 需要落地行动时追加决策逻辑图 |
| 多机制、多结局、多合并症的知识整理 | 思维导图 | 展示中心问题及并列维度 | 需要做选择时追加决策树 |
| 获益/风险、确定性/可及性、疗效/安全性权衡 | 风险-获益矩阵 | 展示方案取舍和优先级 | 需解释证据来源时追加证据关系图 |

不要把所有内容塞进一张图。流程图超过 20 个节点、决策节点超过 6 个、证据节点超过 12 个或连线明显交叉时，拆成两张图。

## 证据关系图标准

证据关系图回答：“哪些证据如何共同支持、限制或反驳当前结论？”

必须包含：

- 一个中心节点：临床问题或病例决策。
- 证据节点：来自文献台账 CSV 中已纳入的指南、共识、关键研究、安全/监管来源；节点标签必须含 `evidence_id` 或文献简称，如 `[G1] KDIGO 2024`、`[R2] DAPA-CKD`。
- 结论节点：`相对确定`、`倾向支持`、`存在争议`、`证据不足`、`不支持/需避免` 中的一种或多种。
- 连线标签：必须写清关系，如 `指南支持`、`RCT 支持`、`安全限制`、`人群不完全匹配`、`与指南冲突`、`仅摘要补充`；不要使用 `强推荐`、`强烈支持` 等高程度措辞。
- 邻近图例或说明：若节点空间不够，可在图下用小表列出 evidence_id、标题、链接和来源级别。
- Mermaid 白板节点本身不强制嵌入链接；不要为了链接破坏 Mermaid 纯文本结构。链接放在图下注释表、正文核心证据解读或正文引用中。
- 可使用多样十六进制语义色区分病例、指南、研究、安全来源、结论和风险；不要使用 `light-yellow` 这类飞书 callout 色名。

禁止：

- 只画“文献 A -> 文献 B -> 结论”的无意义箭头。
- 把图画成“必须治疗/必须停药”的医嘱路径。结论节点应写为“倾向支持与医生讨论...”或“需谨慎/需补充信息”。

## 决策逻辑图标准

决策逻辑图回答：“在这个病例条件下，临床上如何分支判断？”

使用决策树或流程图：

- 起点是病例当前状态或待回答问题。
- 决策逻辑图必须结合病例实际状况，包括用户提供的症状、检验/影像报告文字、诊断分层、合并症、当前用药、关键缺失信息和安全条件；无真实附件时只基于用户文字和检索证据。
- 决策节点必须是一个可以回答的问题，例如 `eGFR 是否低于禁用阈值？`、`是否存在活动性出血？`、`是否符合研究入组人群？`。
- 每个分支必须有条件标签，不允许只有 `是/否` 而没有医学阈值或条件说明。
- 终点是条件性讨论建议，如 `倾向支持方案 A`、`需 MDT 讨论`、`暂不支持，先补充检查`。
- 高风险、禁忌、急症或必须补充资料的分支用风险色，并在正文安全章节展开。

## 思维导图与病例分析图

思维导图适合解释复杂问题的“维度”，不适合表达严格因果或决策顺序。

- 中心节点写临床问题，而不是泛泛的疾病名。
- 一级分支通常包括：病例因素、指南推荐、关键研究、获益、风险、安全监测、现实约束、缺失信息。
- 深度控制在 2-3 层；长段文字放正文或表格，不塞入节点。
- 每个证据分支至少标注 evidence_id。

## 泳道图

泳道图适合把“医学判断”和“执行责任”分开。

- 每条泳道代表一个真实责任方：主管医生、专科/MDT、药师、检验/影像、患者/家属、院内流程等。
- 每个步骤只属于一个责任方。
- 跨泳道连线必须标注触发条件或交付物，如 `复核肾功能`、`药师审方`、`患者知情同意`。
- 不要把泳道写成阶段名，如 `治疗前`、`治疗中`；阶段可作为节点或表格列。

## 视觉与文字规则

颜色必须表达医学语义，不作装饰：

| 语义 | 建议颜色 |
|---|---|
| 临床问题、主路径、主要判断 | 蓝色 |
| 高质量证据支持、明确获益 | 绿色 |
| 条件判断、证据有限、需补充资料 | 琥珀/黄色 |
| 高风险、禁忌、不支持或警示 | 红色 |
| 外部指南、学会、监管来源 | 紫色 |
| 缺失信息、现实约束、人工确认 | 灰色 |

节点和连线要求：

- 节点标签 1-3 行；复杂数据放正文或表格。
- 行动用动词，证据/系统/疾病状态用名词。
- 决策节点用问题句；结论节点用条件性医学表达。
- 每条边都要有标签，标签说明动作、条件、数据、原文推荐方向或证据关系。
- 主路径清晰可 5 秒读懂；风险/例外路径放右侧或下方。
- 对 Feishu 白板或 Mermaid 输出，优先使用可编辑图元；若只能生成静态图或 Mermaid 代码，仍要保证图下有可点击证据说明表。

## PPT/飞书幻灯片图表特殊规则

飞书 PPT/幻灯片中的 `<whiteboard>` 渲染和飞书文档白板不同：回读 XML 时只返回 whiteboard 坐标，不返回 SVG/Mermaid 内容，因此“回读存在 whiteboard”不能证明图表已经显示。

- PPT/幻灯片的关键证据关系图、决策逻辑图、风险路径图默认优先用原生 `shape`、`line`、`table` 组合绘制。
- 结构化数据图表优先用原生 `<chart>`，必须包含 `chartPlotArea` 与 `chartData`，并检查 `dim1` / `dim2` 数据数量匹配。
- Mermaid whiteboard 仅作为可选增强；若使用，必须用 CDATA 包裹，并提供同页或相邻页的表格/形状兜底，避免渲染失败后页面没有主体图。
- 生成 PPT 后按 `lark-ppt` Skill 回读检查页数、主体元素和布局；whiteboard 内部内容需要实际视觉验收，不能只靠 XML 静态检查。

## Mermaid 白板安全规则

`<whiteboard type="mermaid">` 中只能放纯 Mermaid 文本。飞书白板解析比普通 Markdown Mermaid 更严格，生成前遵守：

- 不要在白板块内写任何 HTML/XML 标签，包括 `<br/>`、`<span>`、`<b>`、`<p>`；节点内容过长时拆成多个短节点，或把细节放到图下注释表。
- 不要在白板块内写 Markdown 代码围栏。
- 不要在 Mermaid `style` 或 `classDef` 中使用飞书 callout 颜色名，如 `#light-yellow`、`light-green`；要么省略样式，要么使用合法十六进制颜色，如 `#FEF3C7`、`#DCFCE7`、`#FEE2E2`。
- 尽量避免在节点标签中使用原始 `<`、`&` 等 XML 敏感字符；用 `小于`、`大于等于` 等文字替代。
- 优先使用单行短标签，例如 `VLS: 一线强效激素`；不要用 `<br/>` 在节点里堆多行说明。
- 若白板图生成失败，必须保留同等信息量的降级图：用表格或有序列表表达节点、分支条件、结论，并说明白板解析失败。

安全写法示例仅用于说明 Mermaid 语法，不是模板图。实际报告必须按当前病例、医生问题、CSV 证据和安全条件重新生成节点与连线，不要照抄示例结构。

```xml
<whiteboard type="mermaid">
flowchart TD
    Start[61岁女性外阴易破]:::case
    Q1{是否有剧烈瘙痒}:::decision
    VLS[倾向VLS: 评估活检和一线治疗]:::action
    GSM[倾向GSM: 讨论局部雌激素]:::action
    Risk[排除恶性或感染风险]:::risk
    Follow[2到4周复评疗效和安全性]:::follow
    Start -->|结合症状和查体| Q1
    Q1 -->|是: 考虑VLS| VLS
    Q1 -->|否: 结合萎缩表现| GSM
    VLS --> Risk
    GSM --> Follow
    Risk --> Follow
    classDef case fill:#DBEAFE,stroke:#2563EB,color:#1E3A8A;
    classDef decision fill:#FEF3C7,stroke:#D97706,color:#78350F;
    classDef action fill:#DCFCE7,stroke:#16A34A,color:#14532D;
    classDef risk fill:#FEE2E2,stroke:#DC2626,color:#7F1D1D;
    classDef follow fill:#EDE9FE,stroke:#7C3AED,color:#4C1D95;
</whiteboard>
```

交付前如果已生成报告 XML，运行 `scripts/validate-report-whiteboards.js --attachments none|present <report.xml>`；有报错时先修复白板块或附件状态问题，不能只留下图注，也不能在无真实附件时保留上传资料分析。

## 交付前检查

- 图表是否回答了一个明确问题，而不是装饰。
- 所有证据节点是否来自文献台账 CSV 或参考文献表。
- 正文提到的关键文献是否带超链接、期刊/来源和级别。
- 图中的每条连线是否有医学含义标签。
- 是否把“证据关系”和“临床流程”混在一张过载图里；若过载，拆成两张。
- 是否把风险、禁忌和证据不足用醒目但克制的方式标出。
- Mermaid 白板块和附件状态是否通过 `scripts/validate-report-whiteboards.js --attachments none|present`；若未通过，已改为纯 Mermaid、删除无附件上传资料表述，或降级为表格/列表图。
