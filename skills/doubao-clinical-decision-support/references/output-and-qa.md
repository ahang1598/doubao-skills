# 输出与复核

本文件只管“怎么交付、交付前查什么”。

## 默认输出

默认创建飞书文档，特别是用户明确提及查指南、文献、病例、多方案比较、药物/检查/手术安全、证据冲突或需要专业报告时。问题正文篇幅较长、包含多个子问题、涉及两种及以上药物/方案比较或医学关系复杂时，默认使用 `response_mode=full_report`。但用户明确要求简短、精简、一段话、只给要点或限制字数时，必须降级为 `response_mode=quick_answer`，缩短检索轮次，在聊天中给出结论、必要风险和可点击来源链接，不创建飞书文档或 CSV，并询问是否需要后续生成完整循证报告。未提出简短要求时，`quick_answer` 只用于特别简单、单一、直接、风险低的问题。若用户明确要求 Markdown、表格、PPT、Word/docx 或不要创建文档，按用户要求输出；其中 PPT/幻灯片、Word/docx 和其他非默认格式的专业模板与创建流程见 `output-format-routing.md`。用户要求 Word/docx 时还必须读取 `word-docx-generation.md`。

`response_mode=full_report` 下，聊天窗口只用于短状态和最终文档入口，不承载完整报告。`response_mode=quick_answer` 下，聊天窗口承载简短循证回答，但不要输出检索过程、证据清单、文献台账、todo 或大段推理。

`response_mode=full_report` 时，飞书文档默认使用已有 `lark-doc` Skill 创建，默认使用 XML 结构；只有用户明确要求 Markdown 时才改用 Markdown。长文档优先先建骨架，再按章节追加或更新，避免一次性生成过长内容导致结构混乱。

创建或更新飞书文档前，必须先在本地生成完整报告 XML/Markdown 并运行 `scripts/validate-report-whiteboards.js --attachments none|present <report>`。校验通过后再创建文档；若之后修复 CSV、URL、正文引用或参考文献，必须重新渲染并更新飞书文档。不要先创建飞书文档再校验 CSV，因为后续 CSV 链接修复不会自动回填到已创建的正文。`response_mode=quick_answer` 不运行飞书 XML 校验。

`response_mode=full_report` 且创建飞书文档并取得文档 ID 后，必须交付 CSV 附件：先进入 CSV 所在目录，用相对路径把同一个 CSV 作为文件附件插入该飞书文档末尾。附件类型必须为 `file`，命令形态使用 `--type file`，结构化参数形态使用 `"type": "file"`。CSV 附件用于下载完整循证检索记录，不替代正文来源链接、核心证据解读和“完整参考文献（含链接）”。附件插入成功后，必须在文档末尾追加一句普通正文：`完整文献检索记录见表格。` 不要把 CSV 全量内容、预览表或交付检查表展开进飞书正文。`response_mode=quick_answer` 不需要 CSV 附件。

附件插入方式：使用已有 `lark-doc` Skill 的文件附件能力，从 CSV 所在目录插入同一个 CSV 文件，文件引用使用相对文件名，不传递或展示 CSV 绝对路径；附件类型必须为 `file`，命令形态使用 `--type file`，结构化参数形态使用 `"type": "file"`。不要在最终报告、PPT、Word 或最终回复中展示插入过程或内部工具名。附件插入成功后，用 `lark-doc` 的追加/插入能力在文档末尾追加固定句子：`完整文献检索记录见表格。` 若插入失败，不要伪装成已插入附件；最终回复可用一句话说明附件插入失败，但不要列出本地 CSV、文献台账或台账目录。附件插入成功时，最终回复可以写：`完整的循证检索记录见文档末尾附录。`

默认飞书文档建议使用以下结构，可按临床问题增删小节：

1. 总结论
2. 病例与临床问题
3. 临床问题分析和解答
4. 证据关系与决策逻辑
5. 指南/共识证据
6. 关键研究证据
7. 风险、安全性与下一步
8. 检索策略与证据覆盖
9. 完整参考文献（含链接）
10. 免责声明

为避免长文档一次性写入超时，默认飞书文档按 3 个 section 规划和写入：

| section | 包含章节 | 写入方式 |
|---|---|---|
| `section-1` | 标题、1 总结论、2 病例与临床问题、3 临床问题分析和解答 | 先用 `lark-doc` 创建文档骨架和首段内容 |
| `section-2` | 4 证据关系与决策逻辑、5 指南/共识证据、6 关键研究证据 | 用 `lark-doc` 的追加或插入能力接在 `section-1` 后 |
| `section-3` | 7 风险、安全性与下一步、8 检索策略与证据覆盖、9 完整参考文献（含链接）、10 免责声明 | 用 `lark-doc` 的追加或插入能力最后写入 |

执行时先在本地生成完整 XML 草稿并运行校验；校验通过后再把同一份内容拆成 3 个 XML section 依次写入。不要为了分段写入跳过完整校验，也不要把 3 段拆成 3 个独立文档。若追加或插入失败，保存对应 section XML 文件，修复后从失败 section 重试。

第 3 节“临床问题分析和解答”必须位于指南/共识证据和关键研究证据之前，用于先帮助医生理解并回答当前临床问题。该节先写“问题总结”，随后按病例和问题自由组织分析，可拆成临床子问题，也可按诊断/鉴别、检查路径、治疗或用药选择、安全边界、下一步行动来写；可以适度引用后文证据编号或简称，如 `[G1] 中国指南`、`[R1] DAPA-CKD`。若写出具体文献名，文献名本身仍必须带可点击链接。不要把该节写成泛泛的证据摘要，不要把主轴写成“证据与病例匹配程度”，也不要固定套用模板小标题。

第 9 节为完整参考文献，标题必须包含“链接”。该节默认只使用 `标题｜来源｜链接` 三列表格，按 `核心证据`、`辅助参考/仅摘要`、`排除/待核验` 分组；所有展示记录必须有可点击原文/摘要链接。除非用户明确要求某种参考文献格式（如 GB/T 7714、Vancouver、AMA），不得改成编号列表、纯 bibliography 或多字段宽表；即使用户要求了某种格式，也需要输出文献链接。literature-log CSV 不在正文中给本地路径，作为文档末尾附件交付。

## 工作台消息纪律

允许的中间消息：

- `我会快速核对 1-2 个权威来源，先给你一个简短循证回答。`
- `正在检索和阅读核心指南/文献，完成后会直接生成飞书文档。`
- `已将本轮检索候选写入文献 CSV，正在筛选和阅读核心原文。`
- `核心证据已整理完，正在生成证据图和飞书文档。`
- `飞书文档创建遇到错误，正在改用可重试 XML 草稿兜底。`

禁止的中间消息：

- 不输出长篇“我将整理文献台账...”或逐条证据流水账。
- 不输出完整文献台账、证据综合草稿、检索式清单或 todo。
- 不把工具返回、harness 过程、内部计划整理成用户可见正文。
- 不在创建飞书文档前把完整报告先发到聊天窗口。
- 不在最终报告、PPT、Word 或最终回复里展示内部工具名称；需要说明检索方法时，只写“指南/共识来源、学术文献来源、药品/监管安全来源、原文阅读”等来源类别。

最终回复格式必须使用 `assets/final-response-template.md`，并按 `response_mode` 分流。

`response_mode=quick_answer` 必须满足：

1. 核心要点回答：1-3 条，直接回答临床疑问，说明确定性和适用条件。
2. 风险与注意点：1-3 条，提示最重要的安全风险、证据边界或需要医生确认的条件。
3. 来源链接：至少 1 个可点击权威来源链接；若未找到可核验来源，明确写证据不足。
4. 完整报告引导：一句话询问是否需要创建完整循证检索报告或飞书文档。
5. 继续补充：按当前病例缺失信息写一句，列出 3-6 个最可能改变判断的缺失项。

`response_mode=full_report` 必须满足：

1. 核心要点回答：1-3 条，每条一句话，必须先根据用户提问、病例情景和实际需求直接回答临床疑问，并说明确定性和适用条件；不要用“详见文档”代替回答。
2. 风险与注意点：1-3 条，简短提示最重要的安全风险、证据边界或需要医生确认的条件。
3. 产物入口：必须是 Markdown 链接，如 `[{{标题}}]({{feishu_doc_url_or_local_file}})`；不能只写标题。
4. 附件说明：若 CSV 附件插入成功，可写“完整的循证检索记录见文档末尾附录。”；文档末尾必须已追加“完整文献检索记录见表格。”；若插入失败，用一句话说明失败原因，不展示命令。
5. 继续补充：按当前病例缺失信息写一句，列出 3-6 个最可能改变判断的缺失项，例如“如果您可以继续补充病例其他信息，包括 {{关键缺失项}}，我可以结合刚刚的循证逻辑继续给您分析。”
6. 若失败：明确写 `{{产物类型}}创建失败`，说明原因，并提供可重试飞书文档 XML、PPT XML、Word 草稿或完整 fallback 内容。

`response_mode=full_report` 下，最终回复没有可点击飞书文档链接/标识，或没有明确失败原因和 XML 兜底时，视为未完成。最终回复不要单独提及本地 CSV、文献台账或台账目录。

若用户要求简短回复或简版输出，先判断问题是否特别简单、直接且风险低。满足时按 `response_mode=quick_answer` 回复：核心要点回答 1-2 条、风险与注意点 1-2 条、来源链接、是否需要完整循证报告的引导和继续补充引导；不要在聊天窗口展开证据清单和长段分析。若不满足，仍按 `response_mode=full_report` 执行，可先发一句短状态说明将按循证流程生成完整飞书文档。

不要用 `📋 查看完整文献台账：{{csv_file_name}}` 这类裸文件名输出，也不要把本地路径作为可见文本输出。最终回复不单独列出本地 CSV、文献台账或台账目录。

创建飞书文档后，必须做一次链接抽查：至少检查总结论、临床问题分析和解答、指南/共识证据、关键研究证据和完整参考文献区是否保留可点击来源链接。若创建后回读内容看不到正文链接，回到 XML/Markdown 更新步骤，把文献名改为普通 `<a href="https://...">来源标题</a>` 或等价富文本链接后再交付；不要把“文末有参考文献”当作正文链接合格。

## 正文来源链接闸门

正文不能出现无链接的来源小标题。以下写法不合格：

`《中国慢性肾脏病患者高血压管理指南（2023年版）》`

`DAPA-CKD 研究显示...`

正确写法是把来源名本身做成链接：

`来源：<a href="{{url}}">中国慢性肾脏病患者高血压管理指南（2023年版）</a>（中华肾脏病杂志，A 级中国指南）`

`<a href="{{url}}">DAPA-CKD</a>（NEJM，A 级来源/高质量 RCT，分区/IF 未核验）显示...`

若某段、小节、表格行或图例引用了具体指南/共识/研究/说明书/监管来源，所在段落或同一表格行必须包含对应可点击链接。不要把来源标题单独写成加粗、书名号、无链接 h2/h3 或列表项。

## 附件状态与病例资料边界

每次生成报告前必须完成 `attachment_state` 判定：

| 状态 | 报告写法 |
|---|---|
| `present` | 可加入“上传资料质量与病例提取”，说明真实附件类型、日期、质量、关键提取、局限性 |
| `none` | 只写“用户提供病例信息/证据资料”；不得出现上传资料、附件、患者资料 OCR、质量分级或影像/报告关键提取 |
| `ambiguous` | 说明未检测到可读取附件；若继续分析，只能基于用户文字和检索证据 |

用户在聊天中粘贴的超声、化验、病历或报告文字属于“用户提供的病例文字信息”，不是上传附件。无附件时默认不专门写“未上传附件”章节，除非用户要求说明资料来源。

## 强制样式

默认飞书文档应图文并茂，但不牺牲医学准确性。复杂病例至少使用 4 个结构化或可视化元素：

- 结论 callout：优先用无序列表分点呈现核心结论、关键依据、适用边界和风险；用浅绿或浅蓝突出总体结论，风险或证据不足用浅黄/浅橙/浅红，关键阈值和限制条件可加粗。
- 病例结构表：把人群、诊断、治疗史、当前用药、安全因素和缺失信息压缩成一屏可读表格。
- 证据关系图和决策逻辑图：默认在飞书文档中插入按本病例现场生成的 Mermaid 白板图；证据关系图展示“病例问题 -> 指南/研究/安全证据 -> 个体化判断”，决策逻辑图结合病例实际状况、用户提供信息、关键缺失项和安全条件。模板只提供插入位置，不要求照抄固定 Mermaid 示例。
- 证据矩阵：用表格比较指南、研究、证据等级、来源级别、病例适配性和局限。
- 方案对比表：适合多个治疗/检查/用药选择时使用。
- 风险分层表：对高危、禁忌、需监测事项使用语义底色。
- 思维导图、风险矩阵或泳道图：适合证据多、角色多、分支多或需要把证据与执行路径同时讲清的场景。

若核心证据达到 3 条以上、存在多个方案、指南冲突或复杂安全判断，默认至少放 2 个互补 Mermaid 白板图：一个解释证据如何支持/限制结论，一个解释临床决策如何分支。具体选择和绘制约束见 `clinical-diagrams.md`。

语义色建议：

| 用途 | 颜色 |
|---|---|
| 核心结论/确定获益 | `light-green` |
| 背景信息/检索说明 | `light-blue` |
| 证据有限/需补充资料 | `light-yellow` |
| 证据冲突/需谨慎 | `light-orange` |
| 高风险/禁忌/紧急讨论 | `light-red` |

不要生成伪医学影像。若需要图像，只能用于非诊断性机制示意、流程图或患者沟通图。

## 图表要求

图表必须服务于临床问题，不能只是装饰。

优先图表组合：

| 临床复杂度 | 推荐图表 |
|---|---|
| 多指南/多研究共同支撑一个结论 | 证据关系图 |
| 多方案、多药物或多检查路径 | 证据关系图 + 决策树/流程图 |
| 指南冲突或证据不足 | 证据冲突图/矩阵 + 条件性决策图 |
| 安全性、禁忌、相互作用、特殊人群 | 决策树 + 风险矩阵 |
| MDT、药师审方、检验复核、患者知情同意等多角色流程 | 泳道图 + 风险表 |
| 机制、合并症、结局维度很多 | 思维导图 + 证据关系图 |

图中硬性规则：

- 证据节点必须来自文献台账 CSV，并显示 evidence_id 或文献简称，如 `[G1] ADA 2025`、`[R2] DAPA-CKD`。
- 图中每条边必须有标签，说明 `支持`、`限制`、`冲突`、`外推`、`安全约束`、`需补充信息` 或具体医学条件。
- 决策节点必须写成可回答的问题；分支必须有医学条件或阈值。
- 摘要证据只能标注为 `辅助/仅摘要`，不能画成核心支持节点。
- 若图中节点不能加超链接，必须紧跟一个小型图例表：`evidence_id、标题、链接、来源/期刊、来源级别、阅读状态`。
- Mermaid 白板内部只检查语法和飞书解析安全，不要求节点内嵌链接；来源链接必须放在同页/相邻的图例表、正文核心证据解读或正文引用中。
- 一张图超过 15 个节点、证据节点超过 10 个、决策节点超过 4 个或线条交叉明显时，拆成多图。
- 默认飞书文档优先使用 Mermaid 白板图；若白板解析失败，再降级为表格图或可编辑飞书图元，并保留同等信息量和图例表。PPT/幻灯片可按 `lark-ppt` 要求优先使用原生形状和连线。

## 正文引用格式

正文提到具体指南、共识、试验、论文、说明书、监管来源或 evidence_id 时，第一次出现必须在同一句或同一表格行给出可点击来源链接，并标注期刊/来源和级别。`[G1]`、`[R1]`、`[1]` 这类编号只能作为辅助索引，不能替代链接。

推荐格式：

`<a href="{{url}}">DAPA-CKD</a>（NEJM，A 级来源/高质量 RCT，分区/IF 未核验）`

`<a href="{{url}}">KDIGO 2024 CKD Guideline</a>（KDIGO，A 级指南/国际权威指南）`

不要只在句末粘贴裸 URL，也不要只在文末列参考文献而正文不链接。方案对比、风险、安全性、下一步建议中的“依据/支持证据”列也必须放可点击文献名或指南名。期刊分区、影响因子或来源级别不可核验时写 `未核验`，不要猜。

写作前置表达规则：结论、图表边标签和方案表默认使用 `相对确定`、`倾向支持`、`可作为备选`、`需谨慎讨论`、`证据不足`、`当前不支持` 等条件性表达。不要把“强推荐”“强烈支持”“强烈建议”等高程度措辞留到校验阶段再改。

## 完整参考文献（含链接）

文末“完整参考文献（含链接）”必须使用表格，不要只用编号列表。该节不放本地 CSV 链接或台账目录；完整检索记录作为文档末尾附件交付。

默认表格格式固定为三列：`标题`、`来源`、`链接`。

- `标题`：完整题名、指南名、共识名、说明书名或监管来源名，不要只写 evidence_id。
- `来源`：期刊/机构/数据库/监管来源 + 年份；必要时可在同一格补充来源级别或“仅摘要/全文”。
- `链接`：必须是可点击锚文本，如 `<a href="{{url}}">原文/摘要</a>` 或 `[原文/摘要]({{url}})`；不要写裸 URL、DOI、PMID 或“链接未返回”。
- 除非用户明确要求 GB/T 7714、Vancouver、AMA 等特定格式，文末参考文献不得使用其他列名、编号列表或传统参考文献段落；即使用户要求了某种格式，也需要输出文献链接。

建议分三组：

1. `核心证据`：已核验官方推荐页、指南必要关键章节、说明书或论文正文，能支撑关键结论。
2. `辅助参考`：只能读到摘要。
3. `排除/待核验`：检索到但未支撑正文结论的代表性记录或待核验记录。

每条展示记录至少满足三列信息：标题、来源、可点击链接。无论是否读到全文，最终展示的每条参考文献都必须有 http(s) 链接；没有链接的资料继续补链或从最终参考文献中移除。

表格要求：

- 节首不放本地 CSV 绝对路径链接；只说明完整检索记录作为文档末尾附件交付。飞书文档、PPT 和 Word 正文不得出现本地 CSV 绝对路径。
- 核心证据、辅助参考、排除/待核验三组表格均使用同一表头：`标题｜来源｜链接`。证据级别、阅读状态、病例适配性和局限性等放在正文核心证据解读，或 `来源` 单元格的简短括号中，不在文末扩列。
- 没有可点击链接的记录不要展示在最终文档表格中，只保留在本地 CSV。
- 不展示 `tool`、`search_context`、内部工具名称或 harness 过程；如需要保留来源过程，用“检索批次/来源类别/筛选类别”表达。
- 飞书正文只保留一个完整参考文献章节；CSV 作为飞书文档末尾附件交付，正文不插入 CSV 原样内容、转换表格、全量预览或本地绝对路径链接。

## 飞书写法

优先使用：`<title>`、`<h1>`、`<h2>`、`<callout>`、`<table>`、`<ul>`、`<ol>`、`<a href="https://...">来源标题</a>`。

禁止在飞书文档 XML 中使用上标标签、转义上标标签、脚注式引用或角标引用。不要用上标、脚注或证据编号承担引用功能；改用正文括号如 `[G1]`、`[1]` 加同句可点击文献名，或在同一表格行放可点击文献名。

`lark-doc` XML 的 citation 组件允许作为引用样式容器使用，但它不能替代正文来源链接：组件内必须包含可点击原文/摘要链接，且文献名或指南名本身要可点击。不要把 citation 组件用成脚注、角标编号或无链接的参考文献占位。

飞书文档美化原则参考已有 `lark-doc` Skill：XML 标签本身不要转义，只转义文本中的特殊字符；表格用于行列数据，callout 用于结论、风险和证据不足，不把每段正文都做成高亮块；长文档可先创建章节骨架，再逐节插入内容。

文档中需要外部入口时，优先使用可点击 `<a>` 或 `<bookmark>`；正文首次提到指南、共识、试验或论文时，文献名本身必须带链接。

使用 `<whiteboard type="mermaid">` 时，白板块内只能放纯 Mermaid 文本。不要在白板块内放 `<br/>`、`<span>`、`<b>`、Markdown 代码围栏或飞书 callout 色名。Mermaid 样式若必须使用颜色，只用合法十六进制色值；更稳妥的做法是省略 `style`，用图下注释和表格表达风险色语义。

表格建议：

- 指南表：指南、机构/年份、来源级别、推荐内容、推荐强度/证据等级、病例适配性、原文定位。
- 论文表：类型、标题、来源/期刊、来源级别、年份、可点击原文/摘要链接、证据级别、推荐强度、分区/影响因子、关键片段、核心结论、病例适配性、局限性。
- 决策表：方案、支持证据、潜在获益、主要风险、监测点、需医生确认。


## 最小 QA

交付前只检查这些硬项：

- `response_mode=quick_answer` 时，最终回复已直接回答问题、提供必要风险和可点击来源链接，并引导是否需要完整循证报告。`response_mode=full_report` 时，已通过已有 `lark-doc` Skill 创建飞书文档；最终回复包含飞书文档链接/标识。若未创建，必须有明确失败原因和可重试 XML 草稿。
- 最终回复开头已根据用户提问、病例情景和实际需求，用 1-3 条核心要点直接回答用户问题；没有把文档链接、过程说明或“详见文档”放在回答前面。
- 若用户要求 PPT/幻灯片，已通过已有 `lark-ppt` Skill 创建对应产物并完成该格式校验；若用户要求 Word/docx，已通过已有 `lark-doc` Skill 的 `office-word` 模块创建真正可打开的 `.docx`，或在该路径不可用时明确说明原因并提供完整 Word/docx 可重试草稿。
- PPT/幻灯片默认 10-12 页；除非用户明确要求简版、单页汇报或工具失败兜底，少于 10 页视为不完整。
- 用户要求 Word/docx 时，不得只输出聊天正文、Markdown 文件或把 Markdown 改名为 `.docx`。
- 聊天窗口没有输出长篇 harness/过程日志、文献台账、证据综合草稿或完整报告正文。
- 可见报告、PPT、Word 和最终回复没有展示内部工具名称、工具调用字段、harness 过程或“检索工具”字样。
- 可见报告、PPT、Word 和最终回复没有展示模板/QA 指令，例如“本节目标”“图表要求”“正文来源写法”“正文引用格式示例”“仅 attachment_state=present 时”“每轮检索结果必须先写入 CSV”等；这些约束只能存在于模板注释、reference 或脚本校验中，不能成为用户可见正文。
- `response_mode=full_report` 时章节顺序正确：总结论在最前；“完整参考文献（含链接）”位于免责声明之前，且标题包含“链接”；该节表格默认只有 `标题`、`来源`、`链接` 三列；CSV 作为文档末尾附件交付，不在正文或最终回复里展示本地路径或目录。
- `response_mode=full_report` 时，飞书文档 XML/Markdown 中没有上标标签、脚注式引用或角标式引用；引用使用正文超链接、允许的 citation 样式组件、普通文本编号和参考文献表。
- `response_mode=full_report` 时，飞书文档正文（总结论、临床问题分析和解答、指南/共识证据、关键研究证据、风险与下一步）中至少有可点击文献/指南链接；不得只有文末参考文献链接。`response_mode=quick_answer` 时，回答正文中也必须有可点击来源链接。
- 临床问题分析和解答位于指南/共识证据和关键研究证据之前，且能针对医生问题拆解分析、结合病例条件和证据边界直接作答；证据关系与决策逻辑也位于指南/共识证据之前，帮助读者先建立整体证据地图。该项只检查完整报告。
- `response_mode=full_report` 时，飞书文档不是纯文字：至少包含分点结论 callout、病例结构表、从 CSV 核心记录生成的结构化证据解读、证据关系 Mermaid 白板图和决策逻辑 Mermaid 白板图。
- 已完成 `attachment_state` 判定。`attachment_state=none` 时，病例结构中没有“上传资料与质量”“影像/报告关键提取”“患者资料 OCR”“附件质量”“质量分级”等表述；用户粘贴的报告文字只写为用户提供病例信息。`attachment_state=present` 时，病例结构中已包含上传资料质量、关键提取信息和局限性；若影像质量不足，已明确标注无法可靠解读和需补充资料。
- 复杂病例或多证据场景至少包含 2 个互补图表；图表节点来自文献台账 CSV，边有医学含义标签。
- 图中出现的指南、论文或安全来源在正文或图下注释表中有可点击链接、期刊/来源和级别。
- 若报告 XML 含 `<whiteboard type="mermaid">`，已运行 `scripts/validate-report-whiteboards.js --attachments none|present <file>`；不得出现白板解析失败后只剩图注的情况。无附件任务必须使用 `--attachments none` 校验。
- 已在创建后抽查正文链接：总结论、指南/共识证据、关键研究证据和完整参考文献区仍有可点击来源链接；若回读发现正文链接丢失，已更新修复。
- 结论可追溯到 literature-log CSV 的核心记录；核心证据已核验官方推荐页或能改变结论的必要关键章节，未在信息足够后继续无目的详细阅读。
- 正文首次提到具体指南、共识、试验、论文、说明书或监管来源时，名称本身带超链接，并标注期刊/来源和级别；编号不能替代链接。
- 所有最终参考文献和筛选记录表格项都有可点击原文/摘要链接；不能只写 DOI、PMID、题名或“链接未返回”。
- 已优先检索和引用适配的中国临床指南/本土共识；若未找到，已说明未检索到可用中国指南并使用国际指南互证。
- 结论、方案表、风险表和图表边标签已使用条件性表达，没有使用“强推荐”“强烈支持”“强烈建议”等高程度措辞。
- 参考文献分为“核心证据”“辅助参考/仅摘要”“排除/待核验”，使用 `标题｜来源｜链接` 三列表格，而不是编号列表。
- `response_mode=full_report` 时，已创建本地 literature-log CSV，且 CSV 不为空、不只有表头；正文不展示 CSV 全量表格，也不显示裸绝对路径。`response_mode=quick_answer` 不要求 CSV。
- `response_mode=full_report` 时，默认飞书文档已把 literature-log CSV 作为 `file` 类型附件插入文档末尾；插入必须从 CSV 所在目录执行，并使用相对文件名引用 CSV，命令形态为 `--type file`，结构化参数形态为 `"type": "file"`。若附件插入失败，最终回复已简短说明，且未列出本地 CSV 或台账目录。
- `response_mode=full_report` 时，最终回复不包含原始 literature-log CSV 的本地链接或台账所在目录链接；CSV 使用 UTF-8 with BOM 模板，默认不要另行生成 Excel 副本。`response_mode=quick_answer` 不创建或展示 CSV。
- 最终回复包含贴合病例的“继续补充”引导，列出 3-6 个最可能改变判断的缺失信息，并按重要性排序；不要机械列所有项目。
- `response_mode=full_report` 时，已运行 `scripts/validate-literature-log.js` 或等价检查；CSV 至少包含编号、检索上下文、标题、来源/年份、类型、摘要/占位、链接和筛选/阅读状态。`response_mode=quick_answer` 不要求该项。
- 已遵守检索与 Fetch 预算：没有无目的地反复搜索、没有把整篇长网页/长 PDF 塞进上下文；若超出默认预算，已在报告中说明是因证据冲突、罕见病、高风险安全问题或指南缺失。
- 文末参考文献数量能覆盖指南/共识、关键研究和安全来源；若引用明显少，已解释证据稀少或检索受限。
- 未使用自媒体、营销资料、问答社区、搜索结果页或不可核验转载作为依据。
- CSV 核心记录的 `evidence_level`、`key_excerpt_or_location`、`appraisal_and_applicability` 和 `use_in_report` 已填写；正文涉及分区或影响因子时，仅在可核验后填写，否则写 `未核验`。
- 已把证据与病例人群、疾病阶段、合并症、当前用药和安全风险逐项适配。
- 药品、安全性和特殊人群问题已查说明书、监管或权威指南。
- 未输出直接医嘱、确诊、排除诊断、停药/换药/手术指令。


若飞书创建失败，也要在正文中正常输出完整回答，不得只报告失败。
