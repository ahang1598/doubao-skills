# 核心流程

本文件只定义端到端执行顺序。证据来源、检索工具顺序、预算和评价标准见 `evidence-standards.md`；附件读取见 `patient-upload-analysis.md`；图表规则见 `clinical-diagrams.md`；文档结构、链接、校验和交付规则见 `output-and-qa.md`。

执行原则：快速完成分流，不反复讨论任务模式；除非用户明确要求简短、精简或限定字数，边界不清时直接使用 `full_report` 并开始工作。

## 1. 判定任务归属

先判断用户最终要获得什么：

| 用户目标 | 路由 |
|---|---|
| 围绕具体患者或明确临床问题，获得诊断鉴别、检查路径、用药/治疗比较、安全性、预后或风险分析 | 继续本 Skill |
| 调研某主题/课题的文献全貌、研究进展、热点、里程碑、综述或引用清单 | 改用 `doubao-medical-literature-search` |
| 患者主要询问上传体检、检验、影像或病理报告的异常含义、复查和就医沟通 | 改用 `doubao-medical-report` |
| 医生把报告作为复杂病例证据，并询问诊疗决策 | 继续本 Skill |

分流只看最终产物。病例仅用于限定文献检索人群、用户最终要的是综述或研究调研时，改用文献检索 Skill。

继续本 Skill 后，用一句话锁定决策问题：

`在 {{病例条件或目标人群}} 下，{{方案/检查/用药/随访/诊疗原则}} 是否适合；与 {{对照方案或常规做法}} 相比，其获益、风险和证据边界是什么。`

## 2. 确定输入模式和交付深度

`task_mode` 只决定如何整理输入：

| task_mode | 判定 | 结构化方式 |
|---|---|---|
| `patient_case` | 有具体患者、病例资料、检查结果、用药、附件，或明确询问“这个患者/病例” | 提取病例要素、附件状态和会改变判断的缺失信息 |
| `clinical_question` | 无具体患者，但属于医生的诊疗原则、鉴别诊断、检查路径、药物安全、治疗比较或指南冲突问题 | 明确目标人群、适用/排除条件、关键风险和问题边界 |

`response_mode` 只决定工作深度与交付方式：

| response_mode | 使用条件 | 交付 |
|---|---|---|
| `quick_answer` | 用户明确要求简短、精简、一段话、只给要点或限定字数；或问题单一、边界清楚且风险低 | 1-2 个高价值检索/核验批次；不创建 CSV 和飞书文档；聊天中给结论、风险和可点击来源 |
| `full_report` | 具体病例；多段背景；多个子问题；两种及以上药物/方案比较；医学关系复杂；个体化决策；高风险；指南冲突；需要多篇证据或正式产物 | 创建 CSV，完成检索、筛选、原文阅读、证据综合、图表和目标文档 |

判定规则：

- 用户明确要求简短、精简或限定字数时，优先使用 `quick_answer`。
- 除上述情况外，出现任一复杂条件就使用 `full_report`。
- 一时无法确定时直接使用 `full_report`，不要在模式判断上消耗检索和交付时间。
- `task_mode` 与 `response_mode` 相互独立：无具体病例也可以做完整报告，具体病例在用户明确要求简短时也可以快速回答。

## 3. 执行附件状态闸门

病例或问题结构化前，必须先设置 `attachment_state`：

| 状态 | 判定标准 | 动作 |
|---|---|---|
| `present` | 当前环境实际存在可读取的图片、PDF、Word、截图、影像文件、报告文件、本地路径或连接器附件 | 读取 `patient-upload-analysis.md`，评估质量并提取可用信息 |
| `none` | 没有真实附件；用户只粘贴报告文字、化验值、超声结论、病历摘要或自然语言描述 | 只按“用户提供的病例文字信息”处理，跳过附件分析 |
| `ambiguous` | 用户说“见附件/已上传”，但当前环境没有可读取文件 | 提示“未检测到可读取附件”，基于现有文字给条件性分析或请补传 |

硬性边界：

- 报告样文字、检查描述、化验数值和病历摘要不等于上传附件。
- `attachment_state=none` 或 `ambiguous` 时，不得声称读取附件、OCR、提取影像/报告或评估附件质量，也不添加对应空章节。
- 仅 `attachment_state=present` 时，报告才可加入“上传资料质量与病例提取”，并说明资料类型、日期、质量、关键发现、决策影响和局限。

## 4. 结构化临床问题

### patient_case

只提取影响当前决策的内容，不把病例原文重写成长摘要：

| 维度 | 内容 |
|---|---|
| 基本信息 | 年龄、性别、体重/BSA、妊娠/哺乳、就诊场景 |
| 疾病状态 | 诊断、分期/分型、严重程度、病程、当前关键矛盾 |
| 病例证据 | 症状体征、实验室、影像/病理文字、基因或评分 |
| 治疗与用药 | 既往治疗、疗效/不良反应、当前用药、过敏史 |
| 安全因素 | 肝肾功能、感染、出血、血栓、心血管风险和合并症 |
| 现实约束 | 可及性、费用、院内路径和患者偏好 |
| 缺失信息 | 会改变方案排序、安全边界或下一步检查的资料 |

缺失信息只分为 `必须补充`、`建议补充`、`可后续补充`。存在必须补充项时，可以给条件性暂定结论，但不能给确定性治疗指令。

### clinical_question

不硬填病例表，只明确：

- 目标人群和临床场景。
- 需要回答的诊疗、检查、用药、安全、预后或指南冲突问题。
- 证据适用条件与排除条件。
- 最可能改变答案的风险因素和 3-6 个待限定信息。

随后形成 1-3 个可检索子问题：

- 治疗、管理、预后问题使用 PICOTS。
- 病因或危险因素问题使用 PECO。
- 诊断问题写清目标疾病、待评价检查、参考标准、诊断性能和误诊/漏诊后果。

## 5. 初始化检索与 CSV

读取 `evidence-standards.md`，按问题复杂度设置 `search_budget`、`fetch_budget` 和停止条件；具体工具顺序及说明书检索规则以该文件为准。

`full_report`：

1. 从模板创建 UTF-8 with BOM 的 literature-log CSV。
2. 先用 `general_search` 查找高质量中国指南、本土共识、国家/学会规范和必要国际指南，核对发布机构、年份/版本、适用人群和原文链接。
3. 指南/共识检索后，必须再用 `scholar_search` 检索相关系统综述、RCT、大型队列、诊断研究或其他高质量关键论文，丰富证据类型并核对指南依据。
4. 完成指南/共识和关键研究两类检索后，再判断是否已直接覆盖问题、关键风险是否清楚、是否存在会改变结论的冲突；信息足够时停止扩展。
5. 每轮返后先把高相关候选写入 CSV，再筛选、定向阅读、绘图或写正文；每轮完成 CSV checkpoint，确认候选有标题、来源和可访问链接。

`quick_answer`：

- 仅进行 1-2 个高价值批次，保留 1-3 个最权威的可点击来源。
- 不创建 CSV，不为了凑轮次扩展检索。

执行过程中只报告一句必要状态；候选列表、初步梳理、todo、证据综合草稿、工具名称和中间推理均作为内部工作内容。

## 6. 筛选、精读和引用映射

`full_report` 按以下顺序执行：

1. 在 CSV 中更新 `publication_type`、`evidence_level`、`status` 和 `appraisal_and_applicability`，不要另建第二张筛选表。
2. 仅当已有信息不足以核对关键推荐、入排标准、效应量、亚组、禁忌或安全边界时，才在 `fetch_budget` 内定向阅读相关章节或表格；现有权威信息已足够时，不再详细阅读整篇指南或论文。
3. 在 `status` 中如实记全文、指南正文、摘要或无法访问状态；仅摘要资料只作为辅助参考，不单独支撑关键疗效或安全结论。
4. 为核心证据补齐 `anchor_text`、`key_excerpt_or_location`、`appraisal_and_applicability` 和 `use_in_report`；`evidence_id` 在候选入库时已分配，不要重新编号。
5. 正文具体来源从 CSV 的 `anchor_text + url` 生成文档超链接；不得只写纯文本题名、编号、DOI 或 PMID。
6. 运行 `scripts/validate-literature-log.js`；失败时先修 CSV，再进入报告生成。

`quick_answer` 只打开 1-2 份最权威来源，够回答即停止；来源名称在聊天中直接做成可点击链接。

## 7. 综合证据并直接回答临床问题

先形成条件性结论，再组织文档。结论确定性统一使用：

- `相对确定`
- `倾向支持`
- `存在争议`
- `证据不足`
- `不支持/需避免`

不得使用“强推荐”“强烈支持”“强烈建议”等高程度措辞，也不得输出替代面诊的确诊、排除诊断、停药、换药或手术指令。

“临床问题分析和解答”必须位于指南/共识和关键研究之前：

1. 先写“问题总结”，用 2-4 句话直接回答医生真正关心的问题。
2. 再结合病例或目标人群，说明哪些信息改变诊断概率、方案排序、安全边界或下一步检查。
3. 对主要方案分别说明临床获益、风险、证据依据、适用限制和会改变结论的缺失信息。
4. 可以引用后文的 `[G1]`、`[R1]` 或研究简称；写出具体来源名称时必须同时提供超链接。
5. 仅 `attachment_state=present` 时引用真实附件发现。

涉及多篇证据、多方案、证据冲突或复杂安全判断时，读取 `clinical-diagrams.md`：

- 证据关系图表达指南、研究和安全来源如何支持、限制或冲突。
- 决策逻辑图结合当前病例条件、关键缺失项和安全门槛绘制。
- 图表节点只能来自病例结构或 CSV 中已记录的证据；无附件时不得出现上传报告或 OCR 节点。

## 8. 生成、校验和交付

`quick_answer`：

1. 用 1-3 条核心要点直接回答。
2. 给出必要风险、适用边界和缺失信息。
3. 提供至少 1 个可点击权威来源。
4. 询问是否需要完整循证报告。

`full_report`：

1. 读取 `output-and-qa.md`；用户指定 PPT、Word、Markdown 或表格时，再读取对应格式路由。
2. 先生成完整本地报告，再按 3 个 section 组织：
   - `section-1`：标题、总结论、病例/临床问题、临床问题分析和解答。
   - `section-2`：证据关系与决策逻辑、指南/共识证据、关键研究证据。
   - `section-3`：风险与下一步、证据覆盖、完整参考文献、免责声明。
3. 默认飞书文档调用前，运行 `scripts/validate-report-whiteboards.js --attachments none|present <report>`；`ambiguous` 按 `none` 校验。
4. 校验通过后，通过 `lark-doc` 创建首段并依次追加后两段；PPT 使用 `lark-ppt`，Word/docx 使用 `lark-doc` 的 `office-word` 模块。
5. 正文首次提到具体指南、共识、研究、说明书或监管来源时，名称必须是可点击链接；文末“完整参考文献（含链接）”按 `output-and-qa.md` 输出。
6. 飞书文档创建后，从 CSV 目录以相对路径把同一 CSV 作为 `file` 类型附件插入文末，参数使用 `--type file` 或 `"type": "file"`，随后追加 `完整文献检索记录见表格。`
7. 若修复 CSV、URL、正文引用或参考文献，必须重新渲染、更新产物并复跑校验。
8. 可回读时抽查总结论、临床问题分析、指南/共识、关键研究和完整参考文献区的可点击链接。

最终回复依次提供：

- 1-3 条贴合当前问题的核心答案。
- 1-3 条必要风险或注意点。
- 可打开的产物入口。
- 3-6 个最可能改变判断的待补充信息，以及继续分析引导。

只有产物已创建并完成 CSV 附件交付，或已明确说明创建/附件失败并提供可重试草稿，才算完成。不要以“稍后生成”“继续整理”或“证据已整理完”结束任务；不要在最终回复中暴露本地 CSV 路径或台账目录。
