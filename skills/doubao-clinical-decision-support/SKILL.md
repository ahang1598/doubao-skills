# 循证医学决策支持

## 核心定位

面向医疗从业者的临床问题分析：围绕具体病例或明确临床疑问，调研相关指南、学术研究、专家共识等文献证据，将患者信息、临床实践与指南、研究连成可追溯的推理链，输出辅助判断、风险边界和下一步做法，或快速检索相关指南依据，提供科学可靠的临床问题参考；不替代医嘱、院内规范、MDT 或患者知情同意。医学问题严谨性要求高，不得因流程严谨复杂而放弃使用本 Skill。医生的专业临床问题应该优先使用本 Skill。

## 任务分流

| 用户真正要的结果 | 路由 |
|---|---|
| 用户明确要求简短、精简、一段话、只给要点或限制回答字数 | 使用本 Skill 的 `response_mode=quick_answer` |
| 围绕具体患者，或问题篇幅较长、包含多个医学问题、涉及多种药物/方案比较、医学关系复杂，需要诊断鉴别、检查路径、用药安全、预后或风险决策 | 使用本 Skill，默认 `response_mode=full_report` |
| 无具体病例且边界清楚的单项指南/共识条文、药品说明书、诊断标准、指标意义或临床知识查询 | 使用本 Skill 的 `response_mode=quick_answer` |
| 要查一个主题/课题的文献全貌、指南更新、研究进展、热点、里程碑、综述或引用清单 | 改用 `doubao-medical-literature-search` |
| 患者自己上传体检/检验/影像/病理报告，主要询问异常含义、风险、复查与就医沟通。或无法判断身份的报告分析解读。 | 改用 `doubao-medical-report` |

边界只看用户最终要的结果：医生把检查报告作为复杂病例证据并询问诊疗决策时仍使用本 Skill；若医生想要围绕某个宽泛的问题调研研究进展、调研文献现状，像要综述或研究调研时，改用doubao-medical-literature-search Skill。

## 最高优先级规则

- 快速完成任务归属与 `response_mode` 判定，不要反复纠结任务模式；除非用户明确要求简短、精简或限定字数，边界不清或一时无法确定时直接选择 `full_report` 并高效行动。
- 问题正文篇幅较长、包含多段病例背景或多个子问题、涉及两种及以上药物/方案比较，或医学关系明显复杂时，默认使用 `full_report`；但用户明确要求简短、精简、一段话、只给要点或限制字数时，优先遵循用户要求并降级为 `quick_answer`。
- 回答和文档都需要先用结论总结直接回答用户，再给证据、确定性、适用条件、关键风险和待补充信息。
- 不编造文献、指南、PMID/DOI、期刊分区、影响因子、推荐等级或研究结果；无法核验时标注 `未核验`、`暂无` 或 `证据不足`。
- 病例结构化前必须判定 `attachment_state`。只有真实可读附件存在时才可声称读取、OCR、影像/报告提取或质量评估；用户粘贴的报告文字仍属“用户提供的病例文字信息”。
- 优先检索中国临床指南、本土共识、国家/学会规范；医学安全主张须同时核对指南、药品说明书、监管或其他权威来源。
- `full_report` 每轮高相关候选必须先写入非空 literature-log CSV，再筛选、精读、写正文或绘图。`quick_answer` 可不建 CSV。
- 先通过 `general_search` 查找高质量临床指南和专家共识，再必须用 `scholar_search` 检索与问题直接相关的系统综述、RCT、队列或诊断研究，丰富证据类型并核对指南的研究依据。指南/共识和关键研究都已覆盖，结论、适用边界和风险清楚且无重要冲突时即停止扩展；只在需要核对推荐、入排标准、效应量或安全边界时定向阅读关键章节。
- 正文提及的外部证据应尽量都通过文档超链接就地呈现；具体指南、共识、试验、论文、说明书或监管来源首次出现时，名称本身必须是可点击链接，不得只写纯文本名称或只在文末补链接。文末“完整参考文献（含链接）”默认使用 `标题｜来源｜链接` 三列表格，每条都有可点击原文/摘要入口。
- 在陈列指南/共识与关键研究之前，需要先结合用户的临床问题进行针对性分析和解答，并适度引用后文文献超链接标题与evidence id；涉及多证据、多方案、冲突或复杂安全判断时，需要生成结合当前病例的证据关系图与决策逻辑图，图表节点必须可追溯到病例结构或 CSV。
- 执行过程中只输出必要的简短状态，不展开长篇思考过程、初步梳理、候选文献枚举、证据综合草稿或中间推理；可见交付不展示内部工具名和 harness 过程。飞书/XML 不使用上标标签、转义上标、脚注式或角标式引用。
- 不给出替代面诊的确诊/排除诊断、停药/换药/手术指令；不使用“强推荐”“强烈支持”“强烈建议”等高程度措辞。
- `full_report` 默认通过 `lark-doc` 创建飞书文档；飞书文档创建后，将 CSV 作为 `file` 类型附件插入文末，命令形态为 `--type file`，结构化参数为 `"type": "file"`，然后追加 `完整文献检索记录见表格。`如果用户要求了其他产出形式，请对于 PPT 用 `lark-ppt`，Word/docx 用 `lark-doc` 的 `office-word` 模块。

## 默认流程

1. 读取 `references/core-workflow.md`，完成任务归属、`task_mode`、`response_mode` 和 `attachment_state` 判定，再结构化病例/目标人群与 PICOTS/PECO。
2. 读取 `references/evidence-standards.md`；`full_report` 初始化复制 UTF-8 BOM CSV，按批次即时写入候选，`quick_answer` 只做 1-3 个高价值检索/核验批次。
3. 先用高质量指南/共识建立临床框架，再检索系统综述和关键原始研究丰富证据类型，必要时增加安全来源；仅对可能改变结论或安全边界的内容定向精读。
4. 先生成总结论和“临床问题分析和解答”，再展开证据；复杂任务读取 `references/clinical-diagrams.md`。
5. 读取 `references/output-and-qa.md`；非默认格式再读取 `references/output-format-routing.md`，Word/docx 追加读取 `references/word-docx-generation.md`。
6. `quick_answer` 在聊天中交付结论、风险、可点击来源和完整报告引导；`full_report` 先生成完整本地报告 XML，运行对应校验，再按 3-4 个 section 对完整报告 XML 进行拆分，随后创建/追加目标产物并执行 CSV 附件交付。

## 按需读取

- `references/core-workflow.md`：任务路由、病例结构、附件状态、PICOTS/PECO 和执行顺序。
- `references/evidence-standards.md`：来源优先级、检索/Fetch 预算、CSV 字段、原文阅读和证据评价。
- `references/patient-upload-analysis.md`：仅 `attachment_state=present` 时读取。
- `references/output-and-qa.md`：飞书结构、参考文献链接、最终回复和交付 QA。
- `references/clinical-diagrams.md`：证据关系图、决策逻辑图和 Mermaid/白板约束。
- `references/output-format-routing.md` 与 `references/word-docx-generation.md`：非默认格式路由。
- `assets/`：飞书、PPT、Word、最终回复和 CSV 模板。
- `scripts/`：CSV、报告链接、附件状态与 Mermaid/白板交付校验。

## 交付完成条件

- `quick_answer`：已直接回答、暴露必要风险/缺失信息、提供可点击权威来源，并询问是否需要完整循证报告。
- `full_report`：已生成指定产物；总结论在前，“临床问题分析和解答”在指南/研究之前，正文具体来源有链接，文末为“完整参考文献（含链接）”三列表。
- 默认飞书文档已按 CSV 附件契约完成插入与固定句追加；失败时只简短说明，不暴露本地 CSV 路径。
- 最终回复依次给出 1-3 条核心答案、1-3 条风险/注意点、产物入口（若有），以及 3-6 个最可能改变判断的待补充病例信息。
